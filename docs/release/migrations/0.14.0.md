# Migration to `0.14.0`

## Summary

`0.14.0` makes one backward-incompatible change: **requiring an observed run
receipt is now the default** for `fsgg-sdd verify` and `fsgg-sdd ship`. A test
obligation whose evidence claims `result: pass` but carries no `observedRun`
receipt â€” which reached `satisfied` on every prior release â€” now no longer
satisfies, and both stages **block** it. This is ADR-0035 stage 3b, "the flip",
decided by a human on [FS.GG.SDD#497](https://github.com/FS-GG/FS.GG.SDD/issues/497).

Under the [pre-1.0 `0.x` carve-out](../versioning-policy.md#pre-10-0x-semantics)
this lands on a **minor** bump. The carve-out relaxes the bump; it never relaxes
the note.

The break is **behavioral**: the F# public surface diff `v0.13.0..HEAD` is empty,
and no `--json` output shape changed. A surface or shape diff cannot see it, which
is exactly why ADR-0035 left the flip to a human decision rather than to the
release classifier. The new `--no-require-observed` opt-out flag needs no
adaptation and is not enumerated as a break here.

## Breaking changes and adaptation steps

| # | Breaking change | Affected contract | Consumer adaptation step |

| 1 | `verify` and `ship` now require an observed run receipt by default; an unobserved `result: pass` test obligation no longer reaches `satisfied` and both stages block it (their exit code for that input flips from success to blocked) | `fsgg-sdd verify` / `fsgg-sdd ship` exit-code contract | Record a receipt with `fsgg-sdd evidence --from-test-report <trx-or-junit>` so the obligation is observed; **or** pass `--no-require-observed` to restore the prior behavior during migration. as **Breaking**, pre-1.0 included. The carve-out puts it on a minor bump; the
migration note stays mandatory (FR-009 / FR-010; `migrationNoteRequired Breaking =

## Â§1 â€” Requiring an observed run is now the default

Through `0.13.0`, requiring an observed run was **opt-in**: `fsgg-sdd verify` and
`fsgg-sdd ship` accepted a `--require-observed` flag (ADR-0035 stage 3,
[FS.GG.SDD#422](https://github.com/FS-GG/FS.GG.SDD/pull/422)), and without it a
test obligation that declared `result: pass` / `synthetic: false` reached
`satisfied` even though nothing had ever run a test. That default is the fail-open
[FS.GG.SDD#350](https://github.com/FS-GG/FS.GG.SDD/issues/350) was filed about.

On `0.14.0` (ADR-0035 stage 3b / [FS.GG.SDD#497](https://github.com/FS-GG/FS.GG.SDD/issues/497))
the default is inverted. `verify` raises `verify.unobservedRequiredTest` and
`ship` raises `ship.unobservedEvidence` for any test obligation that claims a pass
without a matching `observedRun` receipt, and both stages block rather than
report ready. The same fabricated lifecycle that shipped clean on `0.13.0` is now
refused.

### Adapting

There are two paths, and the first is the intended one:

**Record the receipt.** Emit a test report from the workflow that already runs the
tests, then record it against the work item:

```sh
fsgg-sdd evidence --from-test-report artifacts/test-results/<project>.trx
```

The obligation becomes observed, `EvidenceObservedCount` rises, and `verify` /
`ship` report ready over evidence that a run actually stands behind.

**Or restore the prior behavior, temporarily.** A work item mid-migration that has
not yet adopted receipts is not stopped dead â€” pass the new opt-out flag:

```sh
fsgg-sdd verify --no-require-observed
fsgg-sdd ship --no-require-observed
```

This reproduces the pre-`0.14.0` behavior byte-for-byte. The legacy
`--require-observed` flag remains recognized as a now-redundant explicit accept,
so pre-flip invocations that passed it do not become an unrecognized-option error.

`obligationIsObserved` is `forall`, not `exists`: one hand-asserted pass beside an
observed one leaves the whole obligation unobserved. Partial receipt adoption on a
work item buys nothing â€” record receipts for every required-test obligation, or
opt the item out with `--no-require-observed` until it is ready.
