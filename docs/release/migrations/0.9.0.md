# Migration to `0.9.0`

## Summary

`0.9.0` makes four backward-incompatible changes: it removes one `--json`
command-report field, it turns two previously-passing lifecycle stages into
stages that can now exit `1`, and it stops silently ignoring an unrecognized
option. A fifth item is not a contract break but does require a one-line edit in
any workspace scaffolded before `0.9.0`.

Under the [pre-1.0 `0.x` carve-out](../versioning-policy.md#pre-10-0x-semantics)
these land on a **minor** bump. The carve-out relaxes the bump; it never relaxes
the note.

`0.9.0` also *adds* a great deal — the `fsgg-sdd surface` command, `validate
--markdown`, `--force-color`, `--accept-upstream`, the `ship-verdict.json`
generated view, and new `work-model.json` reference fields. Additions need no
adaptation and are not enumerated here.

## Breaking changes and adaptation steps

| # | Breaking change | Affected contract | Consumer adaptation step |
|---|---|---|---|
| 1 | Removed `specification.unresolvedAmbiguityCount` (JSON) and the `unresolvedAmbiguities` line (text) | `command-report` (`--json` / `--text`) | Read `clarification.blockingAmbiguityCount` if you were gating; see §1. It is a **different report block**. |
| 2 | `tasks` can now exit `1` with `missingDisposition` | exit-code contract | Re-run `fsgg-sdd tasks` to re-derive the graph, or restore the dropped disposition in `tasks.yml`. See §2. |
| 3 | `plan` can now exit `1` with `stalePlanSnapshot` | exit-code contract | Review the recorded plan decisions against the changed sources, then re-run `fsgg-sdd plan --accept-upstream`. See §3. |
| 4 | **Every** command can now exit `1` with `unknownOption` | exit-code contract | Correct any misspelled or misplaced option. The diagnostic's `correction` lists the options that command accepts. See §4. |

`docs/release/versioning-policy.md` classes both "remove a public field" and
"change an exit-code contract" as **Breaking**. The removed field is signalled
in-band: it takes `command-report`'s `reportVersion` across a **major** boundary
(`1.3.0` → `2.0.0`), so a `--json` consumer can version-gate on `reportVersion`
rather than sniffing for the absent key (FS-GG/FS.GG.SDD#198).

## §1 — The removed counter: a cross-block move, and the replacement is *not* the obvious one

The removed field was on the **`specification`** block. There is no counter on
that block any more. The two counters that exist live on the **`clarification`**
block, which is `null` on every report emitted before `clarify` has run.

Pick by what you actually wanted:

| You wanted | Read | Does it gate? |
|---|---|---|
| The ambiguities that **block** stage advancement | `clarification.blockingAmbiguityCount` | **Yes** — this is the gate |
| Open ambiguities, blocking or not | `clarification.remainingAmbiguityCount` | No — reporting only |
| The ambiguities the spec itself records | `specification.ambiguityIds` (its length) | No |

**Do not assume `remainingAmbiguityCount` is the gate.** It is projected into the
report and nothing else. The only counter read by a gate is
`blockingAmbiguityCount` (`EarlyStageAuthoring.fs`, `LintEngine.fs`). Choosing
`remainingAmbiguityCount` for a gating decision would repeat, in a new field,
exactly the defect that got `unresolvedAmbiguityCount` deleted.

In `--text` the corresponding lines are `remainingAmbiguities:` and
`blockingAmbiguities:`, both emitted under `clarify`.

### Why it was removed rather than fixed

`unresolvedAmbiguityCount` was computed by a regex over `spec.md`'s **own body**,
so it counted ambiguity *mentions in the specification text* — never the
lifecycle's recorded ambiguities. It could report `unresolvedAmbiguities: 4`
while `remainingAmbiguities: 0` and `blockingAmbiguities: 0`, which is the
incoherence that motivated its deletion (feature 093, FS-GG/FS.GG.SDD#164).

It gated nothing, so no consumer loses a decision it was entitled to make.

### Adapting

```diff
- const blocked = report.specification.unresolvedAmbiguityCount > 0;
+ const blocked = (report.clarification?.blockingAmbiguityCount ?? 0) > 0;
```

The `?.` matters: `clarification` is `null` on every report before `clarify`
runs, whereas `specification` was populated from `specify` onward. A consumer
that read the old key defensively (`?? 0`) now silently reads `0` — audit for
that rather than assuming the default is harmless.

## §2 — `tasks` can now exit `1` (`missingDisposition`)

`fix(162)` (FS-GG/FS.GG.SDD#168) moved the disposition-completeness check upstream
into the `tasks` stage. `missingDisposition` is a blocking `DiagnosticError`, and
it was not reachable from `tasks` in `0.8.0`.

A freshly re-derived graph never trips it. It fires when an **authored** edit to
`tasks.yml` dropped a disposition that is still live in the spec/clarify/checklist
/plan facts. Previously such a workspace passed `tasks` and blocked later at
`analyze`; now it fails fast.

**Adaptation:** re-run `fsgg-sdd tasks` to re-derive the graph, or restore the
dropped disposition. The diagnostic names the concrete ids.

## §3 — `plan` can now exit `1` (`stalePlanSnapshot`)

Feature 090 (FS-GG/FS.GG.SDD#163) gave `plan` ownership of its Source Snapshot.
Advancing to `plan` freezes the spec/clarify/checklist authoring window; a later
upstream edit now requires an explicit re-baseline. `stalePlanSnapshot` is a
blocking `DiagnosticError`.

A workspace that edited `spec.md` after planning exited `0` on `0.8.0` and exits
`1` on `0.9.0`.

**Adaptation:** review the recorded plan decisions against the changed sources,
then re-run with the new flag:

```sh
fsgg-sdd plan --accept-upstream
```

(The sibling `planAuthoringWindow` diagnostic is `DiagnosticInfo` and changes no
outcome or exit code. Only `stalePlanSnapshot` blocks.)

## §4 — Every command can now exit `1` (`unknownOption`)

Fix 196 (FS-GG/FS.GG.SDD#196) closes a hole in `fsgg-sdd`'s argument parsing: an
option no command recognized was **silently dropped**, and the command proceeded
with its defaults.

```console
$ cd /tmp/a && fsgg-sdd init --project-root /tmp/b   # 0.8.0
$ echo $?
0
```

The real flag is `--root`. On `0.8.0` that invocation wrote a full SDD skeleton —
`.fsgg/` plus 16 `fs-gg-sdd-*` skill trees across three agent roots — into `/tmp/a`,
left `/tmp/b` empty, and reported `outcome: succeeded`, `diagnostics: []`, exit `0`.
A consumer of the `--json` contract could not distinguish "the flag I passed was
honored" from "the flag was dropped and the default was used".

On `0.9.0` the residue blocks. One `unknownOption` `DiagnosticError` is emitted per
unclaimed token, the command exits `1`, and **nothing is written**:

```console
$ cd /tmp/a && fsgg-sdd init --project-root /tmp/b   # 0.9.0
$ echo $?
1
```

```json
{
  "id": "unknownOption",
  "severity": "error",
  "message": "Unknown option '--project-root' for command 'init'.",
  "correction": "Use one of: --root, --json, --text, --rich, --force-color, --dry-run, --explain, --help, -h. Did you mean '--root'?",
  "relatedIds": [ "--project-root" ]
}
```

(The apostrophes are `'` in the emitted bytes — the serializer's standard escaping,
unchanged by this fix. They are shown unescaped here for legibility.)

### What is recognized

Recognition is **per command**. `--root`, the three format flags, `--force-color`,
`--dry-run`, `--explain`, and `--help`/`-h` are accepted everywhere; every other
option belongs to the commands whose `--help` advertises it. Passing a real flag to
the wrong command — `fsgg-sdd tasks --accept-upstream`, `fsgg-sdd init --provider x` —
is now `unknownOption` rather than a silent no-op.

An unrecognized token wins over `--help`, mirroring the existing rule for an unknown
*command* (FR-011): `fsgg-sdd verify --bogus --help` reports `unknownOption`, not help.
No information is lost — the correction enumerates the options that command accepts.

### Adaptation

Run your invocations once and read the exit code. Anything that was quietly ignored
now names itself. Two shapes to look for:

- **A misspelling of a real flag** — `--project-root`, `--dryrun`, `--work-id`. The
  correction suggests the intended flag when one is close.
- **A flag borrowed from another command.** It never did anything; delete it.

Nothing that exited `0` on `0.8.0` *and* was actually honored changes behavior. Only
invocations that were silently misinterpreted move from `0` to `1`.

## §5 — Not a contract break, but an adaptation is required: `ship-verdict.json` and `.gitignore`

Feature 092 (ADR-0026) makes `readiness/<id>/ship-verdict.json` a **committed**
durable artifact, and changed the seeded `.gitignore` fragment accordingly:

```diff
- readiness/*/
+ readiness/*/*
+ !readiness/*/ship-verdict.json
```

(`readiness/*/*` is load-bearing: git never descends into an excluded *directory*,
so the negation under `readiness/*/` would be silently inert.)

The seed is **no-clobber**, and `doctor`/`upgrade` reconcile a seeded artifact only
when it is **absent** — the check is file existence, not content. A workspace
scaffolded before `0.9.0` therefore keeps `readiness/*/`, and its committed ship
verdict is **silently never staged**. Neither `doctor` nor `upgrade` will tell you.

**Adaptation:** apply the diff above to your workspace `.gitignore` by hand, then
`git add readiness/*/ship-verdict.json`.
