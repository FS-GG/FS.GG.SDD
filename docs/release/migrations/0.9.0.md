# Migration to `0.9.0`

## Summary

`0.9.0` removes one field from the `--json` command-report contract:
`specification.unresolvedAmbiguityCount` (and its `--text` counterpart,
`unresolvedAmbiguities`). Affected: any consumer that reads that key off a
`CommandReport`. No persisted schema, generated-view shape, or CLI surface
changes; every other report field, key order, stream, and exit code is unchanged.

Under the [pre-1.0 `0.x` carve-out](../versioning-policy.md#pre-10-0x-semantics)
this breaking change lands on a **minor** bump. The migration note is still
mandatory — the carve-out relaxes the bump, never the note.

## Breaking changes and adaptation steps

| Breaking change | Affected contract | Consumer adaptation step |
|---|---|---|
| Removed the `specification.unresolvedAmbiguityCount` field (JSON) and the `unresolvedAmbiguities` line (text) | `command-report` (`--json` / `--text`) | Read `clarification.remainingAmbiguityCount` — note it is on a **different report block**. See below. |

### This is a cross-block move, not a rename

The removed field was on the **`specification`** block. Its nearest replacement,
`remainingAmbiguityCount`, is on the **`clarification`** block. A consumer cannot
simply rename the key in place — it must read a different block, one that is
`null` until `clarify` has run.

Pick by what you actually wanted:

| You wanted | Read |
|---|---|
| The open ambiguities that gate stage advancement | `clarification.remainingAmbiguityCount` |
| The subset that *blocks* | `clarification.blockingAmbiguityCount` |
| The ambiguities the spec itself records | `specification.ambiguityIds` (its length) |

In `--text`, the corresponding lines are `remainingAmbiguities:` and
`blockingAmbiguities:`, both emitted under `clarify`.

### Why it was removed rather than fixed

`unresolvedAmbiguityCount` was computed by a regex over `spec.md`'s **own body**,
so it counted ambiguity *mentions in the specification text* — never the
lifecycle's recorded ambiguities. It could report `unresolvedAmbiguities: 4`
while `remainingAmbiguities: 0` and `blockingAmbiguities: 0`, which is the
incoherence that motivated its deletion (feature 093, FS-GG/FS.GG.SDD#164).

There is no behavioural gap to backfill. `remainingAmbiguityCount` is the counter
that always gated stage advancement; `blockingAmbiguityCount` remains the subset
that blocks. The removed field gated nothing, so no consumer loses a decision it
was entitled to make.

### Adapting

```diff
- const open = report.specification.unresolvedAmbiguityCount;
+ const open = report.clarification?.remainingAmbiguityCount ?? 0;
```

The `?.` matters: `clarification` is `null` on every report emitted before
`clarify` runs, whereas `specification` was populated from `specify` onward. A
consumer that reads the old key defensively (`?? 0`, or ignoring unknown keys)
now sees `undefined` where it previously saw a number, and will silently read
`0` — check for that rather than assuming the default is harmless.

### Not a drift-guard exemption

No drift guard caught this removal, and that is **not** the same as the policy
exempting it. `ReleaseContract.fs`'s `jsonInventory` freezes only the *top-level*
report keys — `specification` is frozen as a key, but the counters nested inside
it are not enumerated — and `schema-reference.md` enumerates block names only. The
removal is Breaking under the policy regardless of what the guards happen to
observe. (Widening `jsonInventory` to enumerate nested report keys, so a future
removal reddens a test rather than relying on review, is tracked separately.)

## Also in this release (not breaking)

Everything else merged since `v0.8.0` is additive and needs no adaptation:
`fsgg-sdd surface` (features 086/087), `validate --markdown` + force-color
(feature 088), the feature-shaped `specify` seed (089), the plan Source Snapshot
and `--accept-upstream` (090), the `evidence.yml` always-null omission — which is
parse-compatible in both directions (091) — and the committed compact
`ship-verdict.json` (092, ADR-0026).
