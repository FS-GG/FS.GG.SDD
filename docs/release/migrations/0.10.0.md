# Migration to `0.10.0`

## Summary

`0.10.0` makes two backward-incompatible changes: it changes the `kind` and
`ownership` labels the `--json` command-report reports for the seven lifecycle
artifacts, and it changes the shape of the derived task graph ‚Äî a work item with
`n` requirements now derives `n` implementation tasks instead of `2n`.

Under the [pre-1.0 `0.x` carve-out](../versioning-policy.md#pre-10-0x-semantics)
these land on a **minor** bump. The carve-out relaxes the bump; it never relaxes
the note.

Neither change removes a name from the F# public surface ‚Äî `PublicSurface.baseline`
had no deletions across this release. Both breaks are output-shape / derived-graph
changes only. Additions elsewhere in `0.10.0` need no adaptation and are not
enumerated here.

## Breaking changes and adaptation steps

| # | Breaking change | Affected contract | Consumer adaptation step |
|---|---|---|---|
| 1 | The seven lifecycle artifacts now report `kind: "hybridArtifact"` / `ownership: "hybrid"` (were `"authoredSource"` / `"authored"`) | `command-report` (`--json`) `changedArtifacts[]` | Match lifecycle artifacts on `"hybridArtifact"` / `"hybrid"`. See ¬ß1. |
| 2 | A work item with `n` requirements now derives `n` implementation tasks, not `2n` ‚Äî a `PD-###` plan-decision task subsumed by a requirement task is folded, not dropped | task graph (`fsgg-sdd tasks`) | Re-run `fsgg-sdd tasks` to re-derive the graph; drop any `evidence.yml` declaration whose `subject.id` names a folded task. See ¬ß2. |

`docs/release/versioning-policy.md` classes both a change to a public `--json`
output shape and a change to the derived task-graph shape as **Breaking**, pre-1.0
included. The carve-out puts them on a minor bump; the migration note stays
mandatory (FR-009 / FR-010; `migrationNoteRequired Breaking = true`).

---

## ¬ß1 ‚Äî The lifecycle artifacts relabel: `authoredSource`/`authored` ‚Üí `hybridArtifact`/`hybrid`

The `--json` command-report's `changedArtifacts[]` array records, for each artifact
a command touched, a `kind` and an `ownership`. On `0.9.0` the seven lifecycle
artifacts ‚Äî the `spec.md`/`clarifications`/`checklist`/`plan`/`tasks`/`analysis`
/`verify` sources SDD authors and re-derives ‚Äî reported `kind: "authoredSource"`
and `ownership: "authored"`.

On `0.10.0` those same seven artifacts report `kind: "hybridArtifact"` and
`ownership: "hybrid"` (FS-GG/FS.GG.SDD#312). The label now states the truth these
artifacts always had: they are seeded and re-derived by the tool **and** hand-edited
by an author ‚Äî a hybrid of both, not purely authored source.

### Adapting

A consumer that matched lifecycle artifacts on the old labels must match the new
ones:

```diff
- const isLifecycle = a.kind === "authoredSource" && a.ownership === "authored";
+ const isLifecycle = a.kind === "hybridArtifact" && a.ownership === "hybrid";
```

No artifact changed which of the seven it is; only the two labels moved. A consumer
that ignored `kind`/`ownership` is unaffected.

## ¬ß2 ‚Äî The task graph derives `n` tasks from `n` requirements, not `2n`

On `0.9.0` a work item with `n` requirements derived `2n` implementation tasks: one
requirement task per requirement **and** a separate `PD-###` plan-decision task whose
references were already carried by that requirement task.

On `0.10.0` (FS-GG/FS.GG.SDD#319) a `PD-###` plan-decision task whose refs are
**subsumed** by a requirement task is **folded into** that requirement task rather
than emitted as its own node. A work item with `n` requirements therefore derives
`n` implementation tasks. The folded task's obligation is not lost ‚Äî it is carried
by the requirement task's entry.

A graph re-derived by `fsgg-sdd tasks` on `0.10.0` never contains the folded
`PD-###` id. A workspace whose `evidence.yml` still names that id by `subject.id`
now names a task that does not exist in the re-derived graph.

### Adapting

Re-run `fsgg-sdd tasks` to re-derive the graph, then reconcile evidence:

```sh
fsgg-sdd tasks
```

- **Re-derive the graph.** The `n`-task shape is what every downstream stage now
  reads.
- **Remove the stale evidence declaration.** An `evidence.yml` declaration whose
  `subject.id` names a folded `PD-###` task must be removed; its obligation is now
  carried by the requirement task's entry. Leaving it in place declares an
  obligation against a task the graph no longer contains.

Nothing that referenced a **requirement** task changes ‚Äî requirement task ids are
unchanged. Only a declaration bound to a subsumed `PD-###` task needs the edit.
