# SDD artifact taxonomy — durable vs regenerable

> **Decisions:** [ADR-0018](https://github.com/FS-GG/.github/blob/main/docs/adr/0018-transient-durable-sdd-artifact-taxonomy.md),
> extended by [ADR-0026](https://github.com/FS-GG/.github/blob/main/docs/adr/0026-committed-compact-ship-verdict.md).
> **Owner:** FS.GG.SDD (the lifecycle owns this convention; consuming repos adopt it).

FS.GG.SDD lifecycle artifacts fall into three classes. **Durable** artifacts are authored
input and the record of record — commit them. **Regenerable** artifacts are machine-emitted
output that `fsgg-sdd` recreates deterministically on demand — do not commit them; ignore
them by directory **role**, not with a per-feature exception list. **Durable generated**
artifacts are machine-emitted, byte-stable, drift-guarded — and **committed**, because their
value is bound to the commit that produced them.

Both machine-emitted lists below are the single machine source of truth's projection: they are
**derived from `docs/release/release-readiness.json`** (every `catalog[]` entry whose
`sourceArtifact.kind` is `generatedView`, partitioned on its `durableGenerated` flag) and pinned
to it by a drift guard, so neither can rot as new generated views ship.

## Durable — keep committed

Authored lifecycle sources and the intentional audit record:

| Artifact | Path | Why durable |
|---|---|---|
| Charter | `work/<id>/charter.md` | authored identity |
| Specification | `work/<id>/spec.md` | authored requirements (FR/AC/US ids) |
| Clarifications | `work/<id>/clarifications.md` | authored decisions (CQ/DEC/AMB) |
| Checklist | `work/<id>/checklist.md` | authored requirements-quality review |
| Plan | `work/<id>/plan.md` | authored technical plan |
| Tasks | `work/<id>/tasks.yml` | authored task graph |
| Evidence | `work/<id>/evidence.yml` | authored evidence declarations |
| Contracts / schemas | `work/<id>/contracts/…` | authored machine contracts |
| Governance audit/evidence | (Governance repo) | **record of record — never swept** |

> Governance commits evidence *as the audit trail by design*. This taxonomy targets
> regenerable process noise only; Governance's durable record is explicitly out of scope for
> any cleanup.

## Durable generated — machine-emitted, and committed

The one class ADR-0018 left implicit, fixed by [ADR-0026]. Every other `readiness/<id>/` view
answers *"what is true now?"* — which regeneration answers perfectly. The ship verdict answers
*"what was true at merge?"*, which regeneration **cannot**: re-running `fsgg-sdd ship` reports
today's disposition, not the merge's. Derived from the release catalog (`generatedView` entries
flagged `durableGenerated`):

```gitignore
readiness/<id>/ship-verdict.json
```

It is a compact (≤ 20-line) projection of `ship.json` carrying the verdict — `disposition.state`,
`disposition.blockingFindingIds`, `verificationReadiness.status`, `generator` — plus
**`sourcesDigest`**, one aggregate SHA-256 over the canonical `sources[]` path→digest pairing. That
digest is what makes the compaction sound: it binds the verdict to the exact authored inputs
without carrying their inventory. `ship.json` itself stays regenerable and ignored.

Elsewhere, SDD ships two more durable-generated artifacts outside `readiness/`:
`docs/release/release-readiness.json` and `.agents/skills/skill-manifest.json`.

[ADR-0026]: https://github.com/FS-GG/.github/blob/main/docs/adr/0026-committed-compact-ship-verdict.md

## Regenerable — do not commit (ignore by role)

Per-work-item generated views, regenerated by `fsgg-sdd refresh` (and the read-only generators
`analyze`/`verify`/`ship`). Derived from the release catalog (`generatedView` entries **not**
flagged `durableGenerated`):

```gitignore
readiness/<id>/work-model.json
readiness/<id>/analysis.json
readiness/<id>/verify.json
readiness/<id>/ship.json
readiness/<id>/governance-handoff.json
readiness/<id>/summary.md
readiness/<id>/agent-commands/<target>/guidance.json
readiness/<id>/agent-commands/<target>/commands.md
readiness/<id>/agent-commands/<target>/skills.md
```

Plus other machine-emitted run output that is not lifecycle-cataloged: test-regenerated
baselines/snapshots, diagnostics/log dumps, and `nuget-cache/`.

The positive convention: **ignore the `readiness/<id>/` role once**, do not re-include per
feature. If a specific piece of evidence must be pinned as a durable proof, pin it explicitly
and rarely (`!readiness/<id>/<proof>` ) — the default is ignore.

ADR-0026 adds the one sanctioned exception to "never re-include": a single rule keyed on the
artifact's **role**, constant in the number of work items. That is not the per-feature exception
list ADR-0018 prohibited (`!specs/<feature>/readiness/**`, once per feature, forever).

## Seeded `.gitignore`

`fsgg-sdd init` (and therefore `scaffold`) seeds this fragment into a fresh product's root
`.gitignore` — whole-file, **no-clobber**: an existing `.gitignore` is never overwritten. So a
scaffolded product ignores regenerable output at birth; an **existing** repo adopts the fragment
by copying it in:

```gitignore
# FS.GG.SDD — regenerable lifecycle output (ADR-0018).
# Per-work-item readiness views are generated by fsgg-sdd and regenerated by
# `fsgg-sdd refresh`. They are transient: ignore by role, never commit, and do
# not re-include per feature. See docs/reference/artifact-taxonomy.md.
# Exception: the compact merge-boundary verdict is durable-generated (ADR-0026).
# `readiness/*/*` (contents, not the directory) is load-bearing — git never
# descends into an excluded directory, so the negation below would be inert.
readiness/*/*
!readiness/*/ship-verdict.json

# Generic build & tooling output.
bin/
obj/
artifacts/
TestResults/
.tmp/
nuget-cache/
```

`doctor` reports the seeded `.gitignore` in its expected-vs-present skeleton set, and `upgrade`
no-clobber re-seeds it when **missing**. Note the limitation: `doctor` checks the file's
**presence**, not that a pre-existing consumer `.gitignore` already contains the SDD rule — when
adopting into an existing repo, copy the fragment above yourself.

## Scope note

This doc and the seed are the **SDD producer slice** of ADR-0018. Collapsing a consuming repo's
per-feature exception list, one-time tree cleanup of already-committed regenerable files, and any
history rewrite are **not** owned here — they are per-repo children of
[FS.GG.SDD#110](https://github.com/FS-GG/FS.GG.SDD/issues/110). History is never rewritten;
already-committed files leave the working tree but remain in git history.
