# Phase 1 Data Model: documented grammar entities

This feature documents existing parser grammars; it introduces no new runtime schema. The
"entities" below are the *documented concepts* the skills/reference must describe accurately.
Each maps to a live parser location (the authority the docs must match).

## E1 — Decision-tag resolution (clarify)

- **What it is**: an `AMB-###` id carried on a `DEC-###` line under `## Decisions` or
  `## Accepted Deferrals`, associating the decision/deferral with the ambiguity it resolves.
- **Fields on a decision line** (all extracted by bare-token regex over the raw line; brackets
  cosmetic): the leading `DEC-###` (record id) + any `CQ-###`, `AMB-###`, `FR-###`, `AC-###`,
  `US-###` ids present.
- **Canonical authored form** (what the tool emits, what to teach):
  `- **DEC-001** [CQ-001] [AMB:AMB-001] [FR-001] [AC-001]: <decision text>`.
- **Resolution rule**: an ambiguity is resolved iff (a) some decision/deferral line carries its
  `AMB-###`, AND (b) it is not left as a *blocking* bullet under `## Remaining Ambiguity`.
- **Authority**: `Clarification.fs` decision/ambiguity parsers; `existingResolutionTextForAmbiguity`.

## E2 — Per-stage front matter (gating vs defaulted)

- **What it is**: the YAML `---` header (or whole-doc scalars for `tasks.yml`/`evidence.yml`)
  each stage artifact carries.
- **Gating fields** (absence → `malformed*FrontMatter` / incomplete): stage-specific — see the
  table in `research.md` R1. Superset across stages: `schemaVersion, workId, stage`, plus
  `sourceSpec` (clarify+), `sourceClarifications` (checklist+), `sourceChecklist` (plan);
  `charter` additionally requires `title, changeTier, status`.
- **Defaulted fields** (template includes, parser defaults): `title`, `changeTier`→`tier1`,
  `status`→stage default, `publicOrToolFacingImpact`.
- **Value vocabularies**: `stage` ∈ {charter, specify, clarify, checklist, plan, tasks, analyze,
  implement, evidence, verify, ship} (closed); `schemaVersion` major = 1; `changeTier`, `status`
  free strings (not enforced).
- **Authority**: `Charter`/`Specification`/`Clarification`/`Checklist`/`Plan`/`Task`/`Evidence`
  front-matter parsers; `Identifiers.fs` (stage vocabulary); `Foundation.fs`
  (`frontMatterIdentityDiagnostics`).

## E3 — Clarification id declaration vs reference

- **Declaration**: a line under `## Decisions` or `## Accepted Deferrals` (for `DEC-###`) or
  `## Clarification Questions` (for `CQ-###`) whose leading id is that id. Two declarations of
  one id → `duplicateClarificationId`.
- **Reference**: any mention of an id in another section — safe.
- **Safe deferral pattern**: declare each `DEC-###` exactly once, in exactly one of Decisions or
  Accepted Deferrals.
- **Authority**: `duplicateScopedDiagnostics` over `decisions @ deferrals`.

## E4 — Source Snapshot digest (checklist/plan/tasks/evidence — NOT clarify)

- **What it is**: an optional `sha256:` on a `## Source Snapshot` line (checklist/plan) or
  `sources.digest` scalar (tasks/evidence).
- **Rule**: captured only when exactly 64 hex chars; used solely for staleness detection; a
  non-conforming placeholder is ignored, never a hard error; never required to author. Not
  present in clarifications at all.
- **Authority**: `Checklist.fs`/`Plan.fs` Source-Snapshot regex; `Foundation.fs` staleness.

## E5 — Seeded skill body + manifest (the contract being edited)

- **Canonical body**: `.claude/skills/<name>/SKILL.md` (embedded resource `SeededSkill.<name>`).
- **Mirror**: `.codex/skills/<name>/SKILL.md` byte-identical (hand-maintained); `.agents` bodies
  materialize downstream only.
- **Manifest**: `.agents/skills/skill-manifest.json` — per-skill canonical-body `sha256`,
  regenerated by `fsgg-sdd registry skill-manifest --write`.
- **Invariants** (drift guards): embedded == `.claude`; `.claude` == `.codex`; manifest sha256 ==
  authored body; committed manifest == fresh generation.
