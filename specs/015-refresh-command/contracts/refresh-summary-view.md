# Contract: `readiness/<id>/summary.md` Generated View

**Status**: Tier 1 contract (new generated view) | **Schema version**: 1 |
**View kind**: `summary` (`GenerationManifest.GeneratedViewKind.Summary`,
already defined)

`summary.md` is the SDD-owned human-readable projection of one work item's
structured readiness data. It is **generated**, never authored, and is **not** a
second source of truth (Constitution VII; FR-005, FR-006, FR-013).

## Output path

```
readiness/<id>/summary.md
```

Produced by `GenerationManifest.expectedSummaryOutputPath workId`.

## Source relationships

The summary projects the structured readiness views for the same work item:

- `readiness/<id>/work-model.json`
- `readiness/<id>/analysis.json`
- `readiness/<id>/verify.json`
- `readiness/<id>/ship.json`
- `readiness/<id>/agent-commands/<target>/guidance.json` (per configured target,
  when applicable)

Each is recorded as a `SourceIdentity` (path, source digest, schema version,
schema status) in the summary's `GenerationManifest`.

## File structure

The file begins with a generated-marker header that mirrors the manifest, then a
human-readable body. The header MUST identify the file as generated and record
its sources and generator, so the file can be traced back to declared sources
(FR-007) and a human is warned not to edit it.

```markdown
<!-- GENERATED by fsgg-sdd refresh — DO NOT EDIT.
     view: summary  schemaVersion: 1  generator: <generator-version>
     sources:
       - readiness/<id>/work-model.json  digest:<sha256>  schema:1(current)
       - readiness/<id>/analysis.json    digest:<sha256>  schema:1(current)
       - readiness/<id>/verify.json      digest:<sha256>  schema:1(current)
       - readiness/<id>/ship.json        digest:<sha256>  schema:1(current)
       - readiness/<id>/agent-commands/claude/guidance.json digest:<sha256> schema:1(current)
     outputDigest: <sha256> -->
```

## Faithfulness rules (machine-checkable)

- **FR-006 / SC-008**: Every lifecycle, currency, diagnostic, outcome, and
  next-action fact in `summary.md` MUST appear in the structured readiness views
  or the authoritative command report. The summary introduces **no** fact that
  is absent from them. Tests assert that the per-view state table, diagnostics
  list, outcome, and next action equal the report's `Refresh.PerViewState`,
  `Diagnostics`, `Outcome`, and `NextAction`.
- The summary is **rendered from** the structured data, not parsed back into it;
  the structured views remain authoritative (FR-013).
- If the structured data required to render the summary is missing, stale, or
  blocked, the summary is **not** rendered from unusable data; the report records
  `refreshUnrenderableSummary` and the summary view state is `Blocked` (US3-3,
  FR-005).

## Determinism

- No clocks, durations, ANSI styling, terminal width, directory enumeration
  order, host path separators, randomness, or absolute host paths.
- Source lists are emitted in declared-source order; digests are SHA-256 hex.
- Three identical refreshes over identical inputs produce a byte-identical
  `summary.md` and `outputDigest` (FR-018, SC-004).
