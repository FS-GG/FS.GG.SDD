# Feature Specification: Transient SDD artifact taxonomy + seeded `.gitignore` for regenerable output

**Feature ID**: 073-transient-artifact-taxonomy
**Branch**: `073-transient-artifact-taxonomy`
**Date**: 2026-07-04
**Roadmap**: advances [#110](https://github.com/FS-GG/FS.GG.SDD/issues/110) (cross-repo umbrella; P2 SDD producer slice)
**Decision**: [ADR-0018](https://github.com/FS-GG/.github/blob/main/docs/adr/0018-transient-durable-sdd-artifact-taxonomy.md) — transient vs durable SDD artifact taxonomy; regenerable output gitignored by role, SDD ships the fragment
**Contract**: none new. Consumes SDD's own `docs/release/release-readiness.json` catalog as the derivation source; reuses `init`'s `AgentGuidanceTarget` seed effect.

## Context

Committed Markdown across the four FS-GG repos is ~473k lines — larger than the
~309k lines of F# source — and ~87% of it is regenerable Spec-Kit / SDD *process
output* that the repos already classify as transient yet commit anyway (measured
2026-07-04, [#110](https://github.com/FS-GG/FS.GG.SDD/issues/110)). The worst
symptom is Rendering's `.gitignore`: it ignores `specs/*/readiness/` as transient,
then re-includes it with **98 per-feature exceptions**, leaving 2,053 committed
readiness files under whack-a-mole maintenance.

ADR-0018 fixes this coherently instead of four divergent local decisions. The
convention for *which SDD process artifacts are durable vs regenerable* is owned by
the SDD lifecycle (this repo). SDD is the **producer at the head of the chain**;
the consuming repos adopt what SDD ships. This feature is **only the SDD producer
slice** — two deliverables:

1. A **canonical durable-vs-regenerable taxonomy** SDD publishes, **derived from the
   release catalog** (not a hand-maintained second source of truth) and drift-guarded.
2. A **seeded `.gitignore`** that makes a scaffolded product ignore regenerable
   `readiness/<id>/` output **at birth**, via the same no-clobber seed mechanism as
   the constitution / early-stage guidance.

The Rendering exception-list collapse, the per-repo tree cleanups, and Governance's
audit-record exclusion are **cross-repo children of #110, not this feature.**

### Reproduced against `HEAD` (per cross-repo discipline)

- **SDD's own footprint is already clean.** `readiness/` holds 11 tracked files
  (small reference artifacts); `specs/*/readiness/` is untracked. This repo does
  **not** have the bloat — it is the producer of the convention, not a victim.
- **The regenerable set is already machine-readable.** `docs/release/release-readiness.json`
  `catalog[]` has 9 `sourceArtifact.kind == "generatedView"` entries, every one a
  `readiness/<id>/…` path (`work-model.json`, `analysis.json`, `verify.json`,
  `ship.json`, `governance-handoff.json`, `summary.md`, and the three
  `agent-commands/<target>/…` views). These are `owner: sdd`, byte-stable, and
  regenerated by `fsgg-sdd refresh`. The 10th catalog entry is a `commandOutput`
  (`CommandSerialization.fs`), not an on-disk artifact. **This is the derivation
  source for the taxonomy's regenerable list.**
- **Durable authored sources are not in the catalog** (the catalog enumerates
  generated views + command outputs, not authored input): `spec.md`, `plan.md`,
  `tasks.yml`, `evidence.yml`, `charter.md`, `clarifications.md`, `checklist.md`,
  authored `contracts/`. These are a **stable, lifecycle-defined enumeration** the
  taxonomy authors directly (they do not drift per-release the way the catalog does).
- **`init` seeds no `.gitignore` today.** `Foundation.fs` seeds `.fsgg/constitution.md`,
  `.fsgg/early-stage-guidance.md`, `AGENTS.md`, `CLAUDE.md` via
  `WriteFile(path, text, AgentGuidanceTarget)` (no-clobber). No `.gitignore` is
  written by any command. This is greenfield seeding over the existing effect.

## Goal

SDD publishes the authoritative durable/regenerable taxonomy and ships the seeded
`.gitignore` fragment, so every scaffolded product ignores regenerable
`readiness/<id>/` output by directory *role* at birth, and the taxonomy cannot drift
from the shipped generated-view catalog.

## Functional Requirements

- **FR-001 — Canonical taxonomy doc.** SDD publishes a first-party durable-vs-regenerable
  taxonomy at `docs/reference/artifact-taxonomy.md`: two labeled classes (durable =
  authored lifecycle sources + authored contracts/schemas + Governance's intentional
  record-of-record evidence; regenerable = per-run `readiness/<id>/` generated views,
  test-regenerated baselines/snapshots, diagnostics/log dumps, `nuget-cache/`, other
  machine-emitted run artifacts), each with concrete example paths. It states the
  positive convention: ignore transient output by directory role, not per-feature
  exception; pin a durable proof explicitly and rarely.

- **FR-002 — Regenerable set derived from the release catalog.** The taxonomy's
  regenerable `readiness/<id>/…` path list is **derived from
  `docs/release/release-readiness.json`** (`catalog[]` entries whose
  `sourceArtifact.kind == "generatedView"`), never hand-maintained. A drift guard
  (FR-005) fails if the doc's regenerable readiness list and the catalog's generated-view
  paths diverge. The durable authored-source list is authored directly (not catalog-derived,
  since the catalog does not enumerate authored input) and is likewise drift-guarded against
  the lifecycle stage definitions where one exists.

- **FR-003 — Seeded `.gitignore` for regenerable output.** `fsgg-sdd init` seeds a
  `.gitignore` fragment that ignores the regenerable `readiness/<id>/`-role output,
  reusing the existing `AgentGuidanceTarget` no-clobber write-kind (same authored-SDD-owned,
  deterministic, no-clobber-on-re-run semantics as the constitution / early-stage guidance).
  No new effect, no new schema. The seeded content is generic and value-agnostic (no
  provider-, product-, or feature-specific paths).

- **FR-004 — Seed is part of the coherent skeleton set.** The seeded `.gitignore`
  artifact is delivered by `scaffold` via the reused `init` effects (never marked
  `generatedProduct` provenance); `refresh` never regenerates it; `doctor` reports it as
  a seeded skeleton artifact when present-vs-expected, and `upgrade` no-clobber re-seeds it
  when **missing** — exactly the `AgentGuidanceTarget` class the constitution and
  early-stage guidance already belong to.

- **FR-005 — Drift guards.** A test pins (a) the taxonomy doc's regenerable readiness list
  to the `release-readiness.json` generated-view paths (FR-002), and (b) the seeded
  `.gitignore` fragment bytes to the authored expected content. Divergence in either fails
  the build, so neither the doc nor the seed can silently rot.

- **FR-006 — Descriptive, additive, backward-compatible.** No generator, schema, gate, or
  command-report semantics change. `init`/`scaffold` gain one more seeded artifact; existing
  skeletons are untouched on re-run (no-clobber). Persisted schemas stay at their current
  versions (scaffold-provenance v1).

## Acceptance Criteria

- **AC-001** — `docs/reference/artifact-taxonomy.md` exists, lists both classes with example
  paths, and states the role-based ignore convention. (FR-001)
- **AC-002** — The taxonomy's regenerable readiness list equals the set of `generatedView`
  `sourceArtifact.path` values in `release-readiness.json`; a drift-guard test enforces
  equality and fails on divergence. (FR-002, FR-005)
- **AC-003** — After `fsgg-sdd init` in a fresh directory, a `.gitignore` exists that ignores
  the regenerable `readiness/<id>/`-role output; the seed is byte-deterministic and unchanged
  on a second `init` (no-clobber). (FR-003)
- **AC-004** — The seeded `.gitignore` is delivered by `scaffold` through the reused `init`
  effects and is **not** recorded under `generatedProduct` provenance; `refresh` does not
  rewrite it. (FR-004)
- **AC-005** — `doctor` lists the seeded `.gitignore` in its expected-vs-present skeleton set,
  and `upgrade` no-clobber re-seeds it when missing (a declined/absent step surfaces residual
  drift, never a false "complete"). (FR-004)
- **AC-006** — A drift-guard test pins the seeded `.gitignore` bytes to the authored content.
  (FR-005)
- **AC-007** — Existing golden/deterministic contracts still pass; no JSON byte, exit code, or
  stream routing changes for any command output. (FR-006)

## Open design decision (resolve in plan)

**Seed shape for `.gitignore`.** `AgentGuidanceTarget` is whole-file no-clobber. A fresh
scaffold starts from an empty dir, so a root `.gitignore` seed lands cleanly; but `init` run
in an **existing** repo (or after a provider's `dotnet new` writes its own `.gitignore`)
already has one, and whole-file no-clobber will skip it — leaving the regenerable rule
un-applied. Candidate mechanisms for the plan to choose between:

1. **Whole-file root `.gitignore`, no-clobber** — simplest; only covers the absent case
   (fresh scaffold at birth, matching ADR-0018's framing). Existing repos adopt the documented
   fragment manually / via a cleanup child.
2. **Dedicated fragment file** (e.g. `.fsgg/gitignore` or `.gitignore.fsgg`) seeded
   no-clobber, referenced by the taxonomy doc — always seeded, but git has no `.gitignore`
   include, so the product's root `.gitignore` still needs one authored line.
3. **Ordering within scaffold** — seed the root `.gitignore` before the provider runs so the
   empty-dir case always wins; document the existing-repo case as adopt-manually.

The plan picks one, states why, and defines the exact seeded bytes + the `doctor`/`upgrade`
integration accordingly. Recommendation to evaluate first: **(1)** for the scaffold-at-birth
path (ADR-0018's primary claim), with the taxonomy doc carrying the fragment for existing repos.
