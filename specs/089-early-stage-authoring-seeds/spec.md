# Feature Specification: Early-Stage Authoring Seeds (feature-shaped specify seed + blocked-clarify skeleton)

**Feature Branch**: `089-early-stage-authoring-seeds`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "Close the two early-stage authoring gaps in `fsgg-sdd` reported by FS.GG.Game SDD workflow-feedback (§WD5 + §WD7), tracked as FS-GG/FS.GG.SDD#174 (blocks FS-GG/FS.GG.Templates#118). Both live in `src/FS.GG.SDD.Commands/CommandWorkflow/EarlyStageAuthoring.fs`; they are SDD-internal `specify`/`clarify` authoring defaults, not Templates-owned scaffold content. §WD7 — feature-shaped specify seed: when `specify` runs with no intent stories, the fallback seeds a *meta* story `US-001` = 'As a maintainer, I can specify {title} after chartering the work item.' plus a single meta `AC-001`/`FR-001`. Ship a feature-shaped placeholder seed instead — a real `As a <user>, I can <capability>` shell derived from the work title/description — keeping the `US-001`/`AC-001`/`FR-001` ids and their cross-references intact so downstream stages still resolve them. §WD5 — emit `clarifications.md` on a blocked clarify: a blocked `clarify` produces `changedArtifacts: 0` and no `clarifications.md`, unlike the other early stages, so the operator authors it from scratch. Emit a `clarifications.md` skeleton on the blocked path too, matching the rest of the lifecycle. Constraints: no versioned-contract change; keep output deterministic; the seeded/blocked artifacts must still parse under the existing stage front-matter and authoring-contract grammars."

## Overview

The two earliest authoring stages of the SDD lifecycle each hand the operator something they immediately throw away — or never hand them anything at all.

`fsgg-sdd specify` requires three labeled `--input` facts (user value, scope, requirement) but does **not** require a story or an acceptance scenario. When those are absent it seeds *meta* placeholders that describe the SDD process rather than the feature being specified: the story reads "As a maintainer, I can specify {title} after chartering the work item," and the acceptance scenario reads "Given a chartered work item, when specify runs with intent, then spec.md is created with stable ids." Both are true statements about `specify` and useless statements about the work item. The operator deletes them wholesale and writes the real ones — which means the seed contributes nothing except the `US-001`/`AC-001` ids and their cross-references.

`fsgg-sdd clarify` is worse: when the source specification declares ambiguities that the invocation does not answer, the command blocks with a `missingClarificationAnswer` error and writes **nothing**. It reports `changedArtifacts: 0` and no `work/<id>/clarifications.md` exists. Every other early stage leaves an artifact behind that the operator can edit; `clarify` alone leaves an empty directory and a diagnostic, so the operator hand-authors the file — and hand-authors its front matter, its `CQ-###` question ids, and its `[AMB:AMB-###]` decision tags — from the reference docs.

This feature closes both gaps. `specify` seeds a **feature-shaped** placeholder — a genuine `As a <user>, I can <capability>` story and a matching Given/When/Then scenario, derived from the work item's own title and user value, carrying no SDD process vocabulary — while keeping the `US-001`/`AC-001`/`FR-001` ids and every cross-reference between them intact so `checklist`, `plan`, and `tasks` still resolve them. `clarify` writes a **truthful blocked-state skeleton** on the blocked path: the questions it derived, the ambiguities that remain unanswered, and a `status` that says `needsAnswers` rather than `clarified`.

Neither change touches a versioned cross-repo contract, and neither changes any command's outcome or exit code.

## Clarifications

### Session 2026-07-08

- Q: Which of `specify`'s fallback seeds are actually reachable? → A: Only the **story** and **acceptance-scenario** fallbacks (and the non-goal fallback). `normalizeSpecificationIntent` reports `user value`, `scope`, and `measurable requirement` as *required* missing facts, so `specify` blocks before the scope/requirement fallbacks can ever render. Only the reachable meta seeds are in scope; the unreachable ones are left alone.
- Q: What facts are available to derive a feature-shaped seed from? → A: The **user value** (`intent.UserValue`, a required `--input` fact, so always present on the seeding path) and the invocation's **title**. Verified against the running CLI: the title is *not* read from the charter — it is this invocation's `--title`, falling back to the work id humanized (a `specify` run without `--title` on work item `demo` titles the spec "Demo"). The user value is therefore the load-bearing derivation input; the title is a weak secondary.
- Q: Does a blocked `clarify` fail to write only because its text is absent? → A: No. Verified against the running CLI: a blocked run writes **nothing at all**, because the shared handler shell drops *every* write effect whenever any diagnostic is a blocking error. "Blocked ⇒ zero writes" is a repo-wide invariant, not a local omission — so emitting the skeleton requires a deliberate, narrow carve-out for exactly this one seed write, not a change to what text is computed.
- Q: If the skeleton lists unanswered ambiguities as blocking, can the operator ever resolve them by re-running `clarify` with answers? → A: **Not today** — and this is the trap the naive fix would ship. Verified against the running CLI: with a hand-placed skeleton, `clarify` answering every ambiguity appends the decisions, reports `outcome: succeeded` while still reporting `blockingAmbiguities: 2`, and then `checklist` blocks (exit 1) on "Blocking ambiguity remains unresolved". The blocking lines survive because the existing-file path only *appends* to Remaining Ambiguity, never retires a line. So the feature must also retire a Remaining Ambiguity line once its ambiguity acquires a decision or accepted deferral (FR-018). Verified sufficient: clearing those lines makes `clarify` report `blockingAmbiguities: 0` and `checklist` succeed.
- Q: Can the blocked-clarify skeleton just reuse the existing `clarificationTemplate` as-is? → A: **No.** With zero answers that template computes `status: clarified` (it only reports `needsAnswers` when some answer is explicitly `stillOpen`) and renders "No blocking ambiguity remains." under Remaining Ambiguity. Emitting it verbatim on the blocked path would write an artifact asserting the work is clarified while the command simultaneously blocks for unanswered ambiguities. The skeleton must instead report `needsAnswers` and enumerate each unanswered ambiguity as remaining and blocking.
- Q: Which blocked `clarify` paths emit the skeleton? → A: The **file-absent** path only — where `clarify` derived questions from a specification that parsed successfully but could not resolve every ambiguity into an answer. Paths where a `clarifications.md` already exists (unsafe-overwrite marker, malformed front matter, identity mismatch) already have an operator-owned file and MUST NOT be overwritten. A specification that itself failed to parse yields no questions to seed, so nothing is written.
- Q: Does emitting the skeleton unblock `clarify`? → A: No. The `missingClarificationAnswer` diagnostic remains a `DiagnosticError` and the command still resolves to its blocking outcome and exit code. The only observable deltas are that `changedArtifacts` becomes 1 and the file now exists. The stage advances only once the operator answers the ambiguities.
- Q: Is re-running `clarify` after the skeleton is written safe? → A: Yes, and it must stay safe. Once the skeleton exists, `clarify` takes the existing-file path; a re-run with no new answers must still block on the unanswered ambiguities (it must not read its own skeleton as evidence of resolution) and must not duplicate the `CQ-###` questions it already wrote.
- Q: Does the skeleton's Remaining Ambiguity *explanation text* matter? → A: **Critically.** Found by running the first implementation: `parseRemainingAmbiguity` classifies a line by scanning its prose for `accepted deferral`/`defer` (⇒ `acceptedDeferral`) and `non-blocking` (⇒ `nonBlocking`). A helpful explanation that merely *names* the operator's options — "provide a concrete decision, an accepted deferral, or an explicit still-open note" — therefore parsed as an accepted deferral, giving the skeleton `BlockingAmbiguityCount: 0` and letting `checklist` **succeed with both ambiguities unanswered**. The explanation must contain neither `defer` nor `non-blocking` (FR-021).

## User Scenarios & Testing *(mandatory)*

### User Story 1 - A blocked clarify leaves an editable, truthful skeleton (Priority: P1)

An operator runs `fsgg-sdd clarify --work <id>` against a specification that declares ambiguities, without supplying answers. The command still blocks — the ambiguities are unresolved and nothing may advance to `checklist` — but instead of an empty work directory it now finds `work/<id>/clarifications.md`. The file carries valid stage front matter with `status: needsAnswers`, the `CQ-###` question ids `clarify` derived from the specification's `AMB-###` ambiguities, and a Remaining Ambiguity section naming each unanswered ambiguity as blocking. The operator edits the file in place rather than reconstructing it from the reference docs.

**Why this priority**: This is the gap the feedback reports as most costly — the operator has to reproduce front matter, `CQ-###` ids, and `[AMB:AMB-###]` tag grammar by hand, and every hand-authored file is a chance to trip an authoring-contract grammar. Nothing else in the feature removes manual authoring work.

**Independent Test**: In a fixture whose `spec.md` declares two ambiguities, run `clarify` with no answers. Confirm the command still reports its blocking outcome with the unresolved-ambiguity diagnostic and the same exit code as before, that `work/<id>/clarifications.md` now exists, that its front-matter `status` is `needsAnswers`, that it declares one `CQ-###` question per ambiguity, and that both ambiguities appear under Remaining Ambiguity as blocking.

**Acceptance Scenarios**:

1. **Given** a parsed specification declaring one or more `AMB-###` ambiguities and no existing `clarifications.md`, **When** `clarify` runs without answering them, **Then** the command writes `work/<id>/clarifications.md`, reports one changed artifact, and still blocks with the unanswered-ambiguity diagnostic at the unchanged exit code.
2. **Given** that same blocked run, **When** the written skeleton is inspected, **Then** its front-matter `status` is `needsAnswers` (never `clarified`), and every ambiguity the specification declared appears in Remaining Ambiguity marked blocking.
3. **Given** that same blocked run, **When** the written skeleton is parsed by the clarification parser, **Then** it parses without error, its declared `CQ-###` question ids are the ids the command derived, and its blocking-ambiguity count equals the number of unanswered ambiguities.
4. **Given** a specification that declares no ambiguities at all, **When** `clarify` runs, **Then** behavior is unchanged from today — the command succeeds and writes a `clarified` artifact.

---

### User Story 2 - Re-running a blocked clarify is safe and still blocks (Priority: P1)

Having received the skeleton, the operator re-runs `clarify` — perhaps answering only some ambiguities, perhaps none. The command reads the existing skeleton as an operator-owned authored artifact, not as its own output: it does not treat the skeleton's own questions as answers, it does not duplicate the `CQ-###` ids it already wrote, and it still blocks while any ambiguity remains unanswered. When the operator finally answers the last one, `clarify` resolves as it always did and points to `checklist`.

**Why this priority**: A skeleton that causes the second run to misbehave — to duplicate questions, to declare itself clarified, or to lose the operator's edits — is worse than no skeleton. Idempotence on re-run is what makes the artifact safe to ship, so it rides on the critical path with User Story 1.

**Independent Test**: Run `clarify` twice with no answers on an ambiguity-declaring fixture. Confirm the second run still blocks with the same diagnostic, that the file's `CQ-###` questions are not duplicated, and that its `status` remains `needsAnswers`. Then run a third time answering every ambiguity and confirm the command resolves and the artifact reaches `clarified`.

**Acceptance Scenarios**:

1. **Given** a skeleton written by a prior blocked `clarify`, **When** `clarify` runs again with no answers, **Then** it still blocks with the unanswered-ambiguity diagnostic and the file's question ids are unchanged and not duplicated.
2. **Given** a skeleton written by a prior blocked `clarify`, **When** `clarify` runs again answering every declared ambiguity, **Then** it resolves, each answered ambiguity's line is retired from Remaining Ambiguity, no ambiguity remains blocking, the next lifecycle action is `checklist`, and a subsequent `checklist` run succeeds.
3. **Given** a `clarifications.md` an operator has edited, **When** `clarify` runs again, **Then** the operator's authored content is preserved and never replaced by a fresh skeleton.
4. **Given** an existing `clarifications.md` that carries the unsafe-overwrite marker, or whose front matter is malformed, or whose work id does not match, **When** `clarify` runs, **Then** behavior is unchanged from today — the command reports its existing diagnostic and does **not** write a skeleton over the file.

---

### User Story 3 - The specify seed reads as the feature, not as the process (Priority: P2)

An operator runs `fsgg-sdd specify --work <id>` with the three required intent facts but no story and no acceptance scenario. The generated `spec.md` seeds `US-001` as a real `As a <user>, I can <capability>` sentence about the work item — derived from its title and its stated user value — and seeds `AC-001` as a matching Given/When/Then scenario. Neither mentions chartering, `specify`, `spec.md`, or stable ids. The operator refines the wording instead of deleting a paragraph about the tool and starting over. Every id and cross-reference the previous seed established is still there: `AC-001` still references `[US-001]` and `[FR-001]`, and each `FR-###` still carries its `(Stories: US-001; Acceptance: AC-001)` trailer, so `checklist` coverage and `plan`/`tasks` back-references resolve exactly as before.

**Why this priority**: This is a quality-of-authoring improvement rather than a removal of manual work — the operator was going to rewrite the story either way. It matters, but a wrong-shaped seed costs an edit, whereas a missing `clarifications.md` costs a hand-authored file.

**Independent Test**: Run `specify` on a chartered fixture supplying only user value, scope, and one requirement. Confirm `spec.md` contains a `US-001` story matching the `As a <user>, I can <capability>` shape whose text contains neither "charter" nor "specify" nor "spec.md"; confirm `AC-001` is a Given/When/Then scenario carrying `[US-001] [FR-001]`; confirm the `FR-001` line still ends with `(Stories: US-001; Acceptance: AC-001)`; and confirm the document parses and downstream `checklist` computes FR→AC coverage over it.

**Acceptance Scenarios**:

1. **Given** a `specify` invocation supplying no story, **When** the specification is generated, **Then** the seeded `US-001` is a feature-shaped `As a <user>, I can <capability>` sentence derived from the work item's title and user value, and its text contains no SDD process vocabulary (no reference to chartering, the `specify` command, `spec.md`, or stable ids).
2. **Given** a `specify` invocation supplying no acceptance scenario, **When** the specification is generated, **Then** the seeded `AC-001` is a Given/When/Then scenario about the work item derived from the same two facts, and it carries the `[US-001] [FR-001]` references.
3. **Given** either seed being used, **When** the specification is generated, **Then** the `US-001`/`AC-001`/`FR-001` ids and every cross-reference between them appear in the same positions the previous seed placed them, and each `FR-###` line retains its `(Stories: US-001; Acceptance: AC-001)` trailer.
4. **Given** a `specify` invocation that **does** supply stories and acceptance scenarios, **When** the specification is generated, **Then** the author's text is used verbatim and no seed is substituted — behavior unchanged from today.
5. **Given** a generated specification using the seeds, **When** `checklist` runs against it, **Then** FR→AC coverage resolves exactly as it did with the previous meta seed.

### Edge Cases

- **A specification declaring ambiguities that `clarify` only partially answers**: the command blocks on the unanswered ones and persists **nothing** — the partial answers are not recorded. This is pre-existing behavior (a blocked run writes nothing, and the existing-file path falls back to the unmodified text), it is unchanged by this feature, and it is correct: it never half-writes an operator's artifact. The operator answers every declared ambiguity in one invocation, or edits the skeleton directly.
- **A blocked `clarify` whose `--input` names an unknown `AMB-###`/`CQ-###`/`FR-###` reference**: the unknown-reference diagnostic is reported unchanged. Because the run still derived questions from a parsed specification and no file exists, the skeleton is still written — the operator gets both the diagnostic and the artifact to correct.
- **A `clarify` whose source specification fails to parse**: no questions can be derived, so no skeleton is written. Behavior is unchanged — the specification diagnostic is reported and the command blocks.
- **A work item whose charter title or user value contains characters significant to the artifact grammars** (colons, brackets, `AMB-###`-shaped text): the seeded story, scenario, and skeleton must still parse under the existing front-matter and authoring-contract grammars, and must not introduce a spurious id reference.
- **An empty or whitespace-only user value**: cannot occur on the seeding path — `specify` blocks first, because user value is a required intent fact.
- **A very long title or user value**: the seed remains a single well-formed line per id; it is never wrapped or truncated into a shape that breaks the `- US-001 (P1): …` / `- AC-001 [US-001] [FR-001]: …` line grammars.
- **Repeated runs**: `specify` run twice on unchanged inputs produces a byte-identical `spec.md`, and a re-run of a blocked `clarify` produces no second copy of the derived questions.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When `specify` generates a specification and the invocation supplied no user story, the system MUST seed `US-001` as a feature-shaped user story of the form `As a <user>, I can <capability>`, whose capability clause is derived from the invocation's supplied user value (with the title as secondary context).
- **FR-002**: The seeded `US-001` MUST NOT contain SDD lifecycle process vocabulary — it MUST NOT refer to chartering the work item, to the `specify` command, to `spec.md`, or to establishing stable ids.
- **FR-003**: When `specify` generates a specification and the invocation supplied no acceptance scenario, the system MUST seed `AC-001` as a Given/When/Then scenario about the work item, derived from the same user value and title, and MUST NOT reference the SDD process vocabulary named in FR-002.
- **FR-004**: The seeded story and scenario MUST preserve the existing id assignment and cross-references exactly: `US-001` retains its `- US-001 (P1): …` line shape, `AC-001` retains its `[US-001] [FR-001]` references, and each `FR-###` line retains its `(Stories: US-001; Acceptance: AC-001)` trailer, so `checklist` coverage and downstream `plan`/`tasks` back-references resolve unchanged.
- **FR-005**: When the invocation **does** supply stories or acceptance scenarios, the system MUST use the author's text verbatim and MUST NOT substitute a seed — unchanged from current behavior.
- **FR-006**: When `clarify` blocks because one or more of the source specification's ambiguities were not answered, and no `clarifications.md` exists for the work item, the system MUST write a `work/<id>/clarifications.md` skeleton and report it as a changed artifact.
- **FR-007**: The written skeleton MUST declare front-matter `status: needsAnswers` whenever any declared ambiguity is unanswered; it MUST NOT declare `status: clarified` on a blocked run.
- **FR-008**: The written skeleton MUST enumerate, under its Remaining Ambiguity section, every declared ambiguity that was not answered, each marked blocking; it MUST NOT assert that no blocking ambiguity remains while the command blocks for unanswered ambiguities.
- **FR-009**: The written skeleton MUST declare one `CQ-###` clarification question per declared ambiguity, tagged to that ambiguity, using the same ids the command derived, and MUST parse without error under the existing clarification front-matter and authoring-contract grammars.
- **FR-010**: Writing the skeleton MUST NOT unblock the command. The unanswered-ambiguity diagnostic MUST remain a blocking error and the command's outcome and exit code MUST be unchanged from current behavior; the only observable deltas are the changed-artifact count and the file's existence.
- **FR-011**: The system MUST write the skeleton only when no `clarifications.md` exists. When one exists — including when it carries the unsafe-overwrite marker, has malformed front matter, or declares a mismatched work id — the system MUST report the existing diagnostic and MUST NOT overwrite the file.
- **FR-012**: When the source specification fails to parse, the system MUST NOT write a skeleton; behavior is unchanged.
- **FR-013**: A re-run of `clarify` over a previously written skeleton MUST NOT duplicate the derived `CQ-###` questions, MUST NOT read the skeleton's own questions as answers, and MUST continue to block while any declared ambiguity remains unanswered.
- **FR-014**: A re-run of `clarify` over a skeleton the operator has edited MUST preserve the operator's authored content.
- **FR-015**: All generated text MUST be deterministic: repeated `specify` runs on unchanged inputs produce a byte-identical specification, and repeated blocked `clarify` runs on unchanged inputs produce a byte-identical skeleton.
- **FR-016**: The seeds and the skeleton MUST NOT change any persisted-artifact schema, MUST NOT alter any versioned cross-repo contract, and MUST NOT introduce a new diagnostic code.
- **FR-017**: A title or user value containing characters significant to the artifact grammars MUST NOT produce a seed or skeleton that fails to parse or that introduces a spurious `AMB-###`/`CQ-###`/`FR-###`/`US-###`/`AC-###` reference.
- **FR-018**: When `clarify` records a concrete decision or an accepted deferral for a declared ambiguity, it MUST retire that ambiguity's line from the artifact's Remaining Ambiguity section, and MUST restore the "no blocking ambiguity remains" statement when the section is left with no unresolved entry. Without this, the skeleton FR-006 writes is unresolvable: the recorded decision lands but the blocking line survives, `clarify` reports success while blocking ambiguities remain, and `checklist` blocks.
- **FR-019**: When `clarify` adds a real entry to a section of an existing artifact whose body is only an empty-state placeholder ("No clarification answers recorded.", "No concrete decisions recorded.", "No accepted deferrals recorded."), it MUST retire that placeholder rather than leave it standing beside the real entry.
- **FR-020**: When no ambiguity remains blocking, `clarify` MUST NOT leave the artifact's front matter asserting `status: needsAnswers`. It MUST rewrite **only** the exact status value the tool itself writes (`needsAnswers`), and only within the leading front-matter block, so an operator who chose a different status keeps it (FR-014).
- **FR-021**: The skeleton's generated Remaining Ambiguity explanation MUST parse as a **blocking** entry. The clarification parser classifies a remaining-ambiguity line by scanning its text for `accepted deferral`/`defer` and `non-blocking`, so the explanation MUST NOT contain those words — naming those resolutions as options would classify the line as one of them and silently drop the ambiguity from the blocking count, letting `checklist` pass with the ambiguity unanswered.

### Key Entities *(include if feature involves data)*

- **Specification Seed**: The placeholder story and acceptance scenario `specify` renders when the author supplies none. Derived from the work item's title and user value; carries the fixed `US-001`/`AC-001` ids and their cross-references to `FR-001`.
- **Blocked Clarification Skeleton**: The `clarifications.md` a blocked `clarify` writes when none exists. Carries `status: needsAnswers`, one `CQ-###` question per declared ambiguity, the answers resolved so far (if any), and every unanswered ambiguity listed as remaining and blocking.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A `clarify` run blocked on unanswered ambiguities leaves a `clarifications.md` on disk and reports one changed artifact, where it previously left none and reported zero — confirmed on a fixture whose specification declares ambiguities.
- **SC-002**: The skeleton a blocked `clarify` writes never asserts `status: clarified` and never asserts that no blocking ambiguity remains; its blocking-ambiguity count equals the number of unanswered declared ambiguities — confirmed on fixtures with one and with several ambiguities, and with a partially answered run.
- **SC-003**: A blocked `clarify` reports the same outcome, the same diagnostics, and the same exit code as before this feature; only the changed-artifact count differs — confirmed by comparing against the pre-feature behavior on the same fixture.
- **SC-004**: The written skeleton parses without error under the existing clarification parser, and a second `clarify` run over it blocks with the same diagnostic without duplicating any `CQ-###` question — confirmed by a two-run fixture.
- **SC-005**: An operator who answers every ambiguity after receiving the skeleton reaches an artifact with no blocking ambiguity remaining and `checklist` as the next lifecycle action, and the subsequent `checklist` run succeeds — confirmed end-to-end (this fails today: `clarify` reports success while `checklist` blocks).
- **SC-012**: No section of a `clarify`-updated artifact shows an empty-state placeholder standing beside a real entry.
- **SC-013**: `checklist` run directly against a freshly seeded skeleton **blocks** on unresolved blocking ambiguity — the skeleton's own remaining-ambiguity entries count as blocking, and its blocking count equals the number of declared ambiguities. (A skeleton whose explanation text parsed as an accepted deferral would let `checklist` pass with every ambiguity unanswered.)
- **SC-014**: Once no ambiguity remains blocking, the artifact's front matter no longer says `status: needsAnswers`; an operator-chosen status other than `needsAnswers` is never rewritten.
- **SC-006**: An existing `clarifications.md` is never overwritten by a skeleton, in each of the unsafe-overwrite, malformed-front-matter, and mismatched-work-id cases — confirmed by fixtures showing byte-identical file contents before and after.
- **SC-007**: A `specify` run supplying no story seeds a `US-001` matching the `As a <user>, I can <capability>` shape, and the seeded sentences carry none of the process terms "charter", "specify", "spec.md", or "stable ids" **of their own**. The prohibition binds the seed's own phrasing, not the author-supplied title and user value it interpolates — a work item legitimately titled "Specify Command" puts "specify" in its own acceptance scenario. Confirmed by asserting against a neutral title and user value, plus an unconditional assertion that the old meta-seed phrases ("after chartering the work item", "when specify runs with intent") appear nowhere.
- **SC-008**: The seeded specification's `US-001`/`AC-001`/`FR-001` ids and their cross-references are unchanged from the previous seed, and `checklist` computes the same FR→AC coverage over the new seed as over the old — confirmed by running `checklist` on a seeded specification.
- **SC-009**: A `specify` invocation supplying its own stories and acceptance scenarios produces text identical to today's, with no seed substituted.
- **SC-010**: Repeated runs produce byte-identical output: `specify` twice on unchanged inputs, and a blocked `clarify` twice on unchanged inputs.
- **SC-011**: No persisted-artifact schema, versioned cross-repo contract entry, or diagnostic code is added, removed, or changed by this feature.

## Assumptions

- **Only the reachable seeds are in scope.** `normalizeSpecificationIntent` treats user value, scope, and at least one requirement as required, so `specify` blocks before its scope and requirement fallbacks can render. Only the story and acceptance-scenario fallbacks are live on the seeding path, and only those are reshaped. The non-goal fallback is left unchanged (it is a genuine default, not a meta placeholder about the process).
- **The user value is the derivation input; the title is weak.** These are the only feature-describing facts present when the seed renders. The title is *not* read from the charter — it is the invocation's own `--title`, falling back to the humanized work id. So a `specify` run without `--title` has only the user value to work with, and the seed must read well in that case. The seed derives from these two facts and invents nothing.
- **Emitting a write on a blocked run is a deliberate carve-out.** The shared handler shell drops every write effect when any diagnostic is blocking. The skeleton is the single, named exception to that invariant; it must be scoped so narrowly that no other command, and no other write (including the generated work model), can slip through it. A blocked `clarify` must still write no `readiness/<id>/work-model.json`.
- **Retiring a resolved ambiguity's line is required, not optional polish.** Without FR-018 the feature is a trap: the skeleton's blocking lines outlive the decisions that answer them, so `clarify` reports success while `checklist` blocks. FR-018 also repairs that latent defect for artifacts an operator authored by hand.
- **`<user>` is a generic role.** No fact available at seeding time names the work item's user, so the seeded story uses a generic role rather than guessing a persona. The story's *capability* clause is what carries the derived, feature-specific content.
- **The blocked skeleton is authored source, not a generated view.** It is written with the same authored-source disposition as the clarification artifact `clarify` writes today; it is the operator's file from the moment it lands, which is why FR-011 forbids overwriting an existing one and FR-014 preserves operator edits.
- **The existing `clarificationTemplate` cannot be reused verbatim** on the blocked path: with zero answers it computes `status: clarified` and renders "No blocking ambiguity remains." Both are false while the command blocks, so the template must derive its status and its remaining-ambiguity list from the *unanswered declared ambiguities*, not only from answers explicitly marked still-open.
- **Blocking behavior is unchanged.** This feature is purely additive to what the blocked path *writes*. It does not relax `missingClarificationAnswer`, does not change outcomes or exit codes, and does not let `clarify` advance the lifecycle without answers.
- **No cross-repo contract flip.** Neither change touches `scaffold-provider`, `scaffold-provenance`, `governance-handoff`, or any registry entry; `registry/dependencies.yml` is untouched. The work resolves as a PR in this repository, which closes FS-GG/FS.GG.SDD#174 and unblocks FS-GG/FS.GG.Templates#118.
- **Existing golden/snapshot fixtures will move.** Any test asserting the current meta story, the current meta acceptance scenario, or a zero-changed-artifact blocked-`clarify` outcome is expected to be updated as part of this feature; those updates are the evidence that the behavior changed.
