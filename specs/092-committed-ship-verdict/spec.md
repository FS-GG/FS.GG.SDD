# Feature Specification: Committed Compact Ship Verdict (`readiness/<id>/ship-verdict.json`)

**Feature Branch**: `092-committed-ship-verdict`

**Created**: 2026-07-08

**Status**: Draft

**Input**: User description: "Emit a committed compact ship verdict per ADR-0026 (extends ADR-0018), tracked as FS-GG/FS.GG.SDD#177 (child of FS-GG/.github#235, from FS.GG.Audio feedback §3.7). `readiness/*/` is gitignored by role, so the `shipReady` verdict *at a given commit* is not in git history — 'was `003` ship-ready when it merged?' is answerable only by re-running the tool, which reports today's disposition, not the merge's. Un-ignoring `ship.json` is not the fix: it is 279 lines / ~7 KB, ~59% pure inventory, the footprint class ADR-0018 removed. Introduce the *durable generated* class: `fsgg-sdd ship` additionally emits `readiness/<id>/ship-verdict.json` (target ≤ 20 lines) carrying `schemaVersion`, `workId`, `stage`, `status`/`readiness`, `disposition.state` + `disposition.blockingFindingIds`, `verificationReadiness.status`, `generator`, and `sourcesDigest` — one aggregate SHA-256 over the canonical `sources[]` digest list. Regenerated by `refresh`; byte-stable; catalogued in `docs/release/release-readiness.json` with a marker that places it in the *durable* table of `docs/reference/artifact-taxonomy.md` (the doc must stay catalog-derived); the `init` `.gitignore` seed and its byte-exact drift guard become `readiness/*/*` + `!readiness/*/ship-verdict.json`. Test the negation — git never descends into an excluded *directory*, so the negation under `readiness/*/` is silently inert. Constraints: `ship.json` stays regenerable and ignored; no cross-repo contract registration; additive adoption."

## Overview

`fsgg-sdd ship` is the merge boundary. Every other `readiness/<id>/` view answers *"what is true
now?"* — a question regeneration answers perfectly, which is exactly why [ADR-0018] ignores them
all by directory role. The ship verdict answers a different question: *"what was true at merge?"*
Regeneration cannot answer it. Re-running `fsgg-sdd ship` against today's tree reports today's
disposition; answering it against a historical commit requires that era's authored sources *and*
an era-compatible CLI. FS.GG.Audio hit this in practice (feedback §3.7): the `shipReady` verdict
at a given commit is simply not in git history.

The naive repair — un-ignore `readiness/<id>/ship.json` — reintroduces what ADR-0018 cured. A real
`ship.json` is **279 lines**, of which the `sources[]` digest inventory alone is ~39% and the
`evidenceDispositions` / `generatedViews` / `governanceCompatibility` inventories a further ~21%.
Committed per work item that is the footprint class ADR-0018 removed. The audit question does not
need the inventory: it needs **the verdict**, plus proof of **which inputs produced it**.

This feature introduces the class [ADR-0018] left implicit — **durable generated**: a compact,
byte-stable, drift-guarded generated artifact that *is* committed. SDD already ships two
(`docs/release/release-readiness.json`, `.agents/skills/skill-manifest.json`); exactly one
lifecycle view joins them. `fsgg-sdd ship` additionally writes `readiness/<id>/ship-verdict.json`,
a projection of `ship.json` that drops *inventory* while adding and dropping no *facts* — the
inventory is replaced by `sourcesDigest`, one aggregate SHA-256 that binds the verdict to the exact
authored inputs. `ship.json` itself stays regenerable and ignored.

Two `.gitignore` adoptions carry the decision: the **seeded** consumer fragment (`readiness/*/*` +
`!readiness/*/ship-verdict.json`) and this repository's **own** dogfood rule
(`specs/*/readiness/*/*` + `!specs/*/readiness/*/ship-verdict.json`), because SDD's readiness views
land under `specs/<feature>/readiness/<work-id>/`. They are two adoptions of one decision; landing
only the first would ship an artifact SDD does not itself commit.

[ADR-0026]: https://github.com/FS-GG/.github/blob/main/docs/adr/0026-committed-compact-ship-verdict.md

## Clarifications

### Session 2026-07-08

- Q: Is the `readiness/*/` → `readiness/*/*` change really load-bearing, or cosmetic tidying? → A: **Load-bearing, and verified against real git.** In a scratch repository holding `readiness/003-demo/{ship.json,ship-verdict.json,verify.json,agent-commands/claude/guidance.json}`, the 0018-era pattern `readiness/*/` plus `!readiness/*/ship-verdict.json` adds **nothing** — git never descends into an excluded *directory*, so the negation is silently inert and the verdict stays ignored while the feature looks done. Replacing it with `readiness/*/*` adds **exactly** `readiness/003-demo/ship-verdict.json` and nothing else; `agent-commands/` stays ignored because `readiness/*/*` matches the directory itself, which git then does not descend into. The same experiment confirms the dogfood variant: `specs/*/readiness/*/*` + `!specs/*/readiness/*/ship-verdict.json` stages only `ship-verdict.json` under `specs/<feature>/readiness/<id>/`, and the root `readiness/<id>/` hand-pinned proofs — matched by no rule — remain committed and untouched.
- Q: Does the existing byte-exact `.gitignore` drift guard catch an inert negation? → A: **No, and one of its assertions goes vacuous.** The guard is `ArtifactTaxonomyTests.fs` — *not* `Drift.fs`, which lists `.gitignore` presence-only. Its byte-exact assertion (seeded file ≡ `Foundation.gitignoreSeedText`) does bind. But its companion `Assert.Contains("readiness/*/", seeded)` is a **substring** check that `readiness/*/*` satisfies trivially — it would pass with the negation absent, present-but-inert, or correct. Byte-equality against a constant cannot tell whether that constant *works*; only git can. So the feature must add a behavioral test that stages a real repository (FR-014), not merely tighten the string assert.
- Q: Where does the `.gitignore` byte-exact drift guard live? → A: `tests/FS.GG.SDD.Commands.Tests/ArtifactTaxonomyTests.fs` (two tests: seeded `.gitignore` ≡ `Foundation.gitignoreSeedText`, and the taxonomy doc's fenced seed fragment ≡ that same constant). `Drift.fs` carries `.gitignore` only as a *path* in `expectedArtifactPaths` (presence-only, for `doctor`/`upgrade` re-seeding) and needs no change.
- Q: The taxonomy doc derives its regenerable list from the catalog by drift guard. What happens when a `generatedView` entry is durable? → A: **The guard inverts and must be re-partitioned, not extended.** `ArtifactTaxonomyTests.catalogGeneratedViewPaths()` collects *every* catalog entry whose `sourceArtifact.kind` is `generatedView` and asserts set-equality with the doc's regenerable `readiness/<id>/…` list. Catalogue `ship-verdict.json` as a `generatedView` — as ADR-0026 §4 requires — and the guard demands it appear in the **regenerable** block, contradicting its durability. The guard must partition the catalog by the new `durableGenerated` flag: regenerable block ≡ `generatedView && not durableGenerated`, durable-generated table ≡ `generatedView && durableGenerated`. Both halves stay catalog-derived; the doc never becomes a hand-maintained second source of truth.
- Q: Can `ship-verdict.json` be catalogued without touching the release-boundary guard? → A: **No.** `ReleaseBoundaryTests`' `T024 the catalog adds no GeneratedViewKind beyond the pre-018 enumerable set` pins the catalog's view kinds to `{WorkModel; Analysis; Verify; Ship; Summary; AgentCommands; GovernanceHandoff}` and fails on anything else — including `GeneratedViewKind.Other "shipVerdict"`, which is not in the set either. T024 is feature 018's *"no scope creep"* guard, asserting that **018** added no view kind. ADR-0026 deliberately adds one. T024 must be amended to admit `ShipVerdict` (its name and comment updated to say which feature introduced it), and `ReleaseReadinessCheckTests`' T019 — "the catalog covers every enumerable kind" — then forces the catalog entry to exist. The two guards remain mutually reinforcing.
- Q: Does `fsgg-sdd refresh` regenerate `ship.json` today, and if not, where does verdict regeneration belong? → A: `refresh` **reports** `ship.json`'s currency and never regenerates it (`downstreamClass` yields `Blocked | Missing | Malformed | Stale | AlreadyCurrent`) — re-running the read-only generators out of lifecycle order would corrupt evidence freshness. The one view `refresh` *does* re-project is `governance-handoff.json`, and only when ship's class is `AlreadyCurrent`, because it is a pure projection over `ship.json`. `ship-verdict.json` is the same shape of thing and belongs in exactly that slot, under exactly that gate. Byte-stability between `ship` and `refresh` follows from both calling one identical pure function over the same `ship.json` text — the way `governanceHandoffEmission` is already shared by both handlers.
- Q: Can the verdict be projected from the existing parsed `ShipView`? → A: Not as it stands. `Ship.parseShipView` **flattens `disposition` to a bare string**, reading only `disposition.state` and discarding `disposition.blockingFindingIds` — a field the verdict must carry. `ShipView` must gain the blocking-finding-id list (an additive field on a parse-only record). The alternative already in the codebase, `HandlersShip.parseShipReadinessFacts`, does extract the blocking ids but is a *report* projection, not an artifact one, and drops `sources[]`. The verdict projects from an extended `ShipView`.
- Q: `sourcesDigest` — an aggregate over the digest *values* alone, or over the path→digest pairing? → A: **Over the pairing.** ADR-0026 words it "one aggregate SHA-256 over the canonical `sources[]` digest list" and fixes the *purpose*: "so a later reader can prove the committed verdict corresponds to the committed sources." Hashing the digest values alone does not bind which input produced which digest — two sources exchanging content would be indistinguishable from no change if their sort order also swapped. The canonical pre-image is therefore one line per source, `<path>|<algorithm>:<value>`, over `sources[]` in its canonical (path-sorted) order, joined with `\n`, hashed with the existing `SchemaVersion.sha256Text`. This follows the established `WorkModel.behaviorModelDigest` idiom (canonicalize a list into a separator-joined string, then hash) rather than inventing a new one, and it satisfies the ADR's purpose clause exactly.
- Q: Does the verdict carry `viewVersion`? ADR-0026's field list omits it. → A: **Yes.** Every sibling generated view carries `viewVersion` beside `schemaVersion`, and it is a *fact* of `ship.json`, not inventory — ADR-0026 §1 says the projection "adds and drops no *facts*; it drops *inventory*." Omitting it would be a fact drop. It costs one line and the ≤ 20-line target still holds exactly (20 lines with an empty `blockingFindingIds`).
- Q: Is the ≤ 20-line target achievable, and what happens when findings block? → A: The ship-ready shape is **exactly 20 lines** under the repo's indented-JSON writer. `blockingFindingIds` renders inline as `[]` when empty; a non-empty array costs one line per id. A diagnostically blocked ship is unaffected: `ship.json` is written **only when no diagnostic blocks** (`not hasBlocking`), so the verdict inherits that gate and is never emitted for such a run. It *is* emitted for a run whose `disposition.state` is `needsShipCorrection` with blocking **finding** ids — precisely the audit case worth committing, and the one case where the file exceeds 20 lines, by exactly the number of blocking findings.
- Q: Is `ship-verdict.json` a cross-repo contract? → A: No. It is an SDD-internal generated view like its siblings — `AdditiveOptional`, no `contractVersion`, absent from `registry/dependencies.yml`. Only `governance-handoff.json` carries a cross-repo `contractVersion`. `verify.json` and `governance-handoff.json` are explicitly **out of scope** for committing (ADR-0026 §1): the former answers "what is true now?", which regeneration answers exactly; committing an SDD-side copy of the latter would place a second source of truth for a Governance verdict into SDD's history.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - The merge-boundary verdict survives in git history (Priority: P1)

A maintainer asks, months later, "was work item `003` ship-ready when it merged, and over which
inputs?" They run `git log -p -- readiness/003-*/ship-verdict.json` and read the answer directly:
the disposition state at that commit, the blocking finding ids if any, the verification-readiness
status, the generator that produced it, and the `sourcesDigest` binding it to the exact authored
sources of that era. No CLI, no network, no era-compatible toolchain.

**Why this priority**: This is the audit gap the feedback reports. Nothing else in the feature
delivers it, and every other requirement exists to make this one sound.

**Independent Test**: On a fixture work item, run `fsgg-sdd ship`. Confirm
`readiness/<id>/ship-verdict.json` exists, that it carries the disposition state, blocking finding
ids, verification-readiness status, generator, and a `sourcesDigest` equal to the aggregate hash of
`ship.json`'s `sources[]`, and that the file is at most 20 lines for a ship-ready item. Then, in a
git repository seeded by `fsgg-sdd init`, confirm `git add -A` stages that file.

**Acceptance Scenarios**:

1. **Given** a work item whose `ship` run writes `ship.json`, **When** `ship` completes, **Then** `readiness/<id>/ship-verdict.json` is written alongside it and reported as a changed artifact and a generated view.
2. **Given** that verdict, **When** it is read, **Then** it carries `schemaVersion`, `viewVersion`, `workId`, `stage`, `status`, `generator`, `sourcesDigest`, `verificationReadiness.status`, `disposition.state`, `disposition.blockingFindingIds`, and `readiness` — every one equal to the corresponding fact in `ship.json`.
3. **Given** that verdict for a ship-ready work item, **When** its lines are counted, **Then** there are at most 20.
4. **Given** a work item whose `ship` run blocks on a diagnostic error, **When** `ship` completes, **Then** neither `ship.json` nor `ship-verdict.json` is written — the verdict inherits the existing write gate exactly.

### User Story 2 - The `.gitignore` negation actually fires (Priority: P1)

A product author scaffolds a workspace with `fsgg-sdd init`, runs the lifecycle to `ship`, and
commits. Exactly one file under `readiness/<id>/` enters the index: `ship-verdict.json`. The
`ship.json`, `verify.json`, `work-model.json`, `analysis.json`, `summary.md`, and every
`agent-commands/<target>/…` file stay ignored. The maintainer of this repository gets the same
behaviour under `specs/<feature>/readiness/<id>/`, and the root `readiness/<id>/` hand-pinned
durable proofs stay committed and untouched.

**Why this priority**: An inert negation ships a feature that looks complete and delivers nothing —
the verdict is written to disk and silently never committed. This is the failure mode ADR-0026 §2
names explicitly, and it is invisible to every byte-exact string guard. It rides the critical path
with User Story 1.

**Independent Test**: In a real temporary git repository seeded by `fsgg-sdd init`, materialize a
full `readiness/<id>/` tree including `agent-commands/<target>/guidance.json`, run `git add -A`, and
assert the staged set under `readiness/` is exactly `{readiness/<id>/ship-verdict.json}`. Repeat for
this repository's own `.gitignore` rule against `specs/<feature>/readiness/<id>/`.

**Acceptance Scenarios**:

1. **Given** a git repository whose `.gitignore` is the one `fsgg-sdd init` seeds, **When** a full `readiness/<id>/` tree is created and `git add -A` runs, **Then** the staged paths under `readiness/` are exactly `readiness/<id>/ship-verdict.json`.
2. **Given** that same repository, **When** the nested `readiness/<id>/agent-commands/<target>/` files are probed, **Then** each is ignored.
3. **Given** this repository's own `.gitignore`, **When** a `specs/<feature>/readiness/<id>/` tree is created and `git add -A` runs, **Then** only `specs/<feature>/readiness/<id>/ship-verdict.json` is staged, and files under the root `readiness/<id>/` remain committed and unignored.
4. **Given** an existing product `.gitignore`, **When** `init` (or `upgrade`) runs, **Then** it is not clobbered — adoption of the amended fragment stays manual, and until adopted the workspace's verdicts stay ignored, degrading to ADR-0018's behavior with nothing broken.

### User Story 3 - `refresh` brings the verdict to currency, byte-identically (Priority: P1)

An author edits an authored source, then runs `fsgg-sdd refresh`. The work model and agent guidance
regenerate; `analysis.json`/`verify.json`/`ship.json` are *reported*, never re-run. When `ship.json`
is already current, `refresh` re-projects `ship-verdict.json` from it and the bytes are identical to
what `ship` wrote. When `ship.json` is stale, missing, malformed, or blocked, the verdict inherits
that class and is not rewritten from a stale input.

**Why this priority**: A generated view that only one command can produce drifts the moment anything
else changes. `governance-handoff.json` already solved this shape; the verdict must not solve it
differently. Byte-stability across the two producers is the whole basis of the drift guard.

**Independent Test**: Run `ship`, capture the verdict's bytes, run `refresh` on an unchanged tree,
and assert the bytes are unchanged and the view reports already-current. Then make `ship.json`
stale and assert `refresh` reports the verdict under ship's inherited class rather than rewriting it.

**Acceptance Scenarios**:

1. **Given** a work item whose `ship.json` is current, **When** `refresh` runs, **Then** `ship-verdict.json` is re-projected, its bytes are identical to those `ship` wrote, and it is reported among the already-current views.
2. **Given** a work item whose `ship.json` is stale, missing, malformed, or blocked, **When** `refresh` runs, **Then** `ship-verdict.json` inherits that class and is not rewritten from the stale input.
3. **Given** the same inputs, **When** `ship` and `refresh` each produce the verdict, **Then** both call one identical pure projection and the outputs are byte-identical.
4. **Given** any two runs on unchanged inputs, **When** the verdicts are compared, **Then** they are byte-identical — no clock, no absolute path, no ANSI.

---

### User Story 4 - The taxonomy doc stays catalog-derived (Priority: P2)

A maintainer adds a future generated view. The regenerable list in
`docs/reference/artifact-taxonomy.md` and its new durable-generated table both continue to be
projections of `docs/release/release-readiness.json`, pinned by drift guard. Neither table can rot,
and neither becomes a hand-maintained second source of truth — the property ADR-0026 §4 fixes as a
requirement while leaving SDD the mechanism choice.

**Why this priority**: The doc is a projection today and must remain one. It is a correctness
property of the documentation system rather than of the shipped artifact, so it trails the three P1
stories — but shipping the catalog entry without re-partitioning the guard makes the build red, so
it is not optional.

**Independent Test**: Add the catalog entry with `durableGenerated: true` and confirm the taxonomy
guard passes with `ship-verdict.json` appearing in the durable table and absent from the regenerable
block. Remove the flag and confirm the guard fails.

**Acceptance Scenarios**:

1. **Given** the release catalog, **When** the taxonomy guard runs, **Then** the doc's regenerable `readiness/<id>/…` list equals exactly the catalog's `generatedView` entries that are **not** `durableGenerated`.
2. **Given** the release catalog, **When** the taxonomy guard runs, **Then** the doc's durable-generated table equals exactly the catalog's `generatedView` entries that **are** `durableGenerated`.
3. **Given** the catalog entry for `ship-verdict.json`, **When** its `sourceArtifact.kind` is read, **Then** it is `generatedView`, and its `durableGenerated` flag is `true`.
4. **Given** every other catalog entry, **When** its `durableGenerated` flag is read, **Then** it is `false` — the field is additive and the existing entries' classification is unchanged.

### Edge Cases

- **A `ship.json` with an empty `sources[]`**: `sourcesDigest` is the hash of the empty canonical pre-image, computed by the same helper. It is a well-defined, stable value, not an omitted field.
- **A work item whose `disposition.state` is `needsShipCorrection` with blocking findings**: the verdict is still written (the write gate is the *diagnostic* blocking gate, not the disposition), it enumerates the blocking finding ids, and it exceeds 20 lines by exactly that count. This is the audit case most worth committing.
- **A `sources[]` entry whose digest algorithm is not `sha256`**: the pre-image carries the algorithm verbatim (`<path>|<algorithm>:<value>`), so a future algorithm change alters the aggregate rather than silently colliding with it.
- **A consuming repo that has not adopted the amended fragment**: its verdicts stay ignored. The seed is no-clobber and never rewrites an existing `.gitignore`; the change degrades to ADR-0018's behavior and nothing breaks.
- **A consuming repo that pins a durable proof under `readiness/<id>/`**: the existing `!readiness/<id>/<proof>` convention still works — `readiness/*/*` excludes contents, so a per-file negation continues to fire. The root `readiness/<id>/` proofs in *this* repository are matched by no rule at all and are unaffected.
- **A `ship-verdict.json` that already exists on disk from a prior run**: it is a generated view, overwritten deterministically. Because it is byte-stable, an unchanged run produces no git diff.
- **`fsgg-sdd validate`'s determinism matrix**: the verdict joins the enumerated generated outputs. A new generated view the harness does not enumerate is a silent coverage hole.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: When `fsgg-sdd ship` writes `readiness/<id>/ship.json`, the system MUST additionally write `readiness/<id>/ship-verdict.json`, and MUST report it as a changed artifact and as a generated view.
- **FR-002**: The verdict MUST carry exactly these facts, each equal to the corresponding fact in the `ship.json` it projects: `schemaVersion`, `viewVersion`, `workId`, `stage`, `status`, `generator`, `sourcesDigest`, `verificationReadiness.status`, `disposition.state`, `disposition.blockingFindingIds`, and `readiness`. It MUST carry no other field — it drops inventory, never facts.
- **FR-003**: `sourcesDigest` MUST be a single SHA-256 over a canonical pre-image built from `ship.json`'s `sources[]` in canonical (path-sorted) order, as one line per source of the form `<path>|<algorithm>:<value>`, joined with `\n`. The pre-image MUST bind each source's path to its digest, so the committed verdict provably corresponds to the committed sources.
- **FR-004**: The verdict MUST be at most 20 lines when `disposition.blockingFindingIds` is empty, and MUST exceed that only by the number of blocking finding ids.
- **FR-005**: When `ship` blocks on a diagnostic error and therefore writes no `ship.json`, the system MUST write no `ship-verdict.json`. The verdict inherits the existing write gate; an incomplete ship is never recorded as a verdict.
- **FR-006**: `fsgg-sdd refresh` MUST re-project `ship-verdict.json` when, and only when, `ship.json`'s currency class is already-current — the same gate under which it re-projects `governance-handoff.json`. When `ship.json` is blocked, missing, malformed, or stale, the verdict MUST inherit that class and MUST NOT be rewritten from the stale input.
- **FR-007**: `ship` and `refresh` MUST produce the verdict through one identical pure projection over the same `ship.json` text, so the two producers are byte-identical by construction.
- **FR-008**: The verdict MUST be byte-stable: repeated runs on unchanged inputs produce byte-identical output, with no clock reading, no absolute path, and no ANSI.
- **FR-009**: The `init` `.gitignore` seed MUST replace the directory rule `readiness/*/` with the contents rule `readiness/*/*` followed by the negation `!readiness/*/ship-verdict.json`, and MUST keep its whole-file, no-clobber `AgentGuidanceTarget` write kind so an existing consumer `.gitignore` is never overwritten.
- **FR-010**: This repository's own `.gitignore` MUST adopt the same decision in its dogfood shape: `specs/*/readiness/*/*` plus `!specs/*/readiness/*/ship-verdict.json`, leaving the root `readiness/<id>/` hand-pinned durable proofs committed and unmatched by any rule.
- **FR-011**: The byte-exact drift guard pinning the seeded `.gitignore` to `Foundation.gitignoreSeedText`, and the guard pinning the taxonomy doc's fenced seed fragment to that same constant, MUST both continue to hold over the amended bytes.
- **FR-012**: `ship-verdict.json` MUST be catalogued in `docs/release/release-readiness.json` with `sourceArtifact.kind: generatedView`, stability `AdditiveOptional`, no `contractVersion`, and an additive `durableGenerated: true` flag. Every pre-existing catalog entry MUST carry `durableGenerated: false`, leaving its classification unchanged.
- **FR-013**: The taxonomy doc's regenerable list MUST equal exactly the catalog's `generatedView` entries that are not `durableGenerated`, and its durable-generated table MUST equal exactly those that are — both pinned by the drift guard, so neither table becomes a hand-maintained second source of truth.
- **FR-014**: The system MUST carry a behavioral test that stages a real git repository and asserts that, under the seeded `.gitignore`, the staged set beneath `readiness/<id>/` is exactly `{ship-verdict.json}` — every sibling view and every nested `agent-commands/<target>/…` file remaining ignored. A byte-exact assertion over the seed constant MUST NOT be treated as evidence that the negation fires.
- **FR-015**: The equivalent behavioral test MUST cover this repository's own dogfood rule over `specs/<feature>/readiness/<id>/`, and MUST assert that the root `readiness/<id>/` proofs stay unignored.
- **FR-016**: The release catalog's view-kind guard MUST be amended to admit the verdict's generated-view kind, and the guard requiring the catalog to cover every enumerable view kind MUST continue to hold — so adding the kind without cataloguing it, or cataloguing it without admitting the kind, both remain build failures.
- **FR-017**: `fsgg-sdd validate`'s determinism matrix and output classification MUST enumerate `ship-verdict.json`, so the new generated view is not a silent coverage hole.
- **FR-018**: The feature MUST NOT change `ship.json`: it remains regenerable, ignored, and unchanged in shape. It MUST NOT commit `verify.json` or `governance-handoff.json`, and MUST NOT register `ship-verdict.json` as a cross-repo contract or add it to `registry/dependencies.yml`.
- **FR-019**: Adoption MUST be additive. A consuming repository that has not copied the amended fragment MUST continue to behave exactly as it does under ADR-0018 — its verdicts stay ignored, and nothing breaks.

### Key Entities *(include if feature involves data)*

- **Ship Verdict**: The compact, committed, durable-generated projection of `ship.json`. Carries the merge-boundary answer (disposition state, blocking finding ids, verification-readiness status), its provenance (`workId`, `stage`, `generator`, `schemaVersion`, `viewVersion`), and `sourcesDigest`. Produced by `ship`, re-projected by `refresh`, never hand-authored.
- **Sources Digest**: One aggregate SHA-256 over the canonical path→digest pairing of `ship.json`'s `sources[]`. Replaces the inventory it summarizes; it is what makes the compaction sound.
- **Durable-Generated Class**: The artifact class ADR-0018 left implicit — machine-emitted, byte-stable, drift-guarded, and committed. Marked on a release-catalog entry by `durableGenerated: true`, from which the taxonomy doc's durable table is derived.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: After `fsgg-sdd ship` on a fixture work item, `readiness/<id>/ship-verdict.json` exists, is reported as a changed artifact and a generated view, and is at most 20 lines for a ship-ready item — where previously no such file existed.
- **SC-002**: Every fact in the verdict equals the corresponding fact in the `ship.json` it projects, and the verdict carries no field outside the FR-002 list — confirmed by projecting a fixture `ship.json` and comparing field by field.
- **SC-003**: `sourcesDigest` equals the SHA-256 of the canonical path→digest pre-image, recomputed independently in the test from the fixture's `sources[]`; changing any single source's path or digest changes the aggregate.
- **SC-004**: In a real temporary git repository seeded by `fsgg-sdd init`, `git add -A` over a full `readiness/<id>/` tree stages exactly `readiness/<id>/ship-verdict.json` and nothing else beneath `readiness/` — including nothing beneath `agent-commands/<target>/`. Under the pre-feature seed the same test stages nothing, proving the negation is load-bearing.
- **SC-005**: In this repository, `git check-ignore` reports `specs/<feature>/readiness/<id>/ship-verdict.json` as unignored, every sibling view as ignored, and every root `readiness/<id>/` proof as unignored.
- **SC-006**: `ship` followed by `refresh` on an unchanged tree leaves `ship-verdict.json` byte-identical, and `refresh` reports it among the already-current views; with a stale `ship.json`, `refresh` reports the verdict under ship's inherited class and does not rewrite it.
- **SC-007**: Two consecutive `ship` runs on unchanged inputs produce byte-identical verdicts; the file contains no timestamp, no absolute path, and no ANSI.
- **SC-008**: A `ship` run that blocks on a diagnostic error writes neither `ship.json` nor `ship-verdict.json` — confirmed on a blocking fixture.
- **SC-009**: The taxonomy doc's regenerable list equals the catalog's non-`durableGenerated` `generatedView` entries, and its durable table equals the `durableGenerated` ones; flipping the flag on the verdict's entry makes the guard fail in a named, comprehensible way.
- **SC-010**: The release-boundary view-kind guard admits exactly the pre-018 set plus the verdict's kind, and the catalog-covers-every-kind guard still passes — adding the kind without a catalog entry fails the build.
- **SC-011**: `fsgg-sdd validate` enumerates `ship-verdict.json` in its determinism matrix and classifies it by basename; a coverage-gap test confirms no generated view is missing from the harness.
- **SC-012**: `ship.json` is byte-identical before and after this feature on the same fixture, and no entry is added to `registry/dependencies.yml`; no `contractVersion` is introduced.
- **SC-013**: A repository whose `.gitignore` still carries the ADR-0018-era `readiness/*/` rule behaves exactly as before — its verdict is written to disk and ignored by git, and no command fails.

## Assumptions

- **The verdict is a projection, not a second computation.** It re-reads the `ship.json` text `ship` just produced (the way `governanceHandoffEmission` already does) rather than recomputing readiness from the work model. This is what makes `ship` and `refresh` byte-identical by construction rather than by coincidence, and it keeps `ship.json` the single source of truth for the facts the verdict carries.
- **`ShipView` gains one additive field.** `Ship.parseShipView` currently discards `disposition.blockingFindingIds`, flattening `disposition` to a bare `state` string. The verdict needs the list, so the parse-only record gains it. This is additive to a public record; it moves the `PublicSurface.baseline` for `FS.GG.SDD.Artifacts`.
- **A new `GeneratedViewKind` case is deliberate scope.** ADR-0026 introduces a first-class generated view, so `GeneratedViewKind` gains `ShipVerdict` and feature 018's T024 guard is amended to say so. Using `Other "shipVerdict"` would evade neither T024 nor the intent; a named case keeps matches checked.
- **`durableGenerated` is SDD's mechanism choice.** ADR-0026 §4 fixes the requirement (the taxonomy doc stays catalog-derived) and names an explicit flag on the catalog entry as "the obvious shape," leaving the choice to SDD. This feature takes it. The flag is additive to `SchemaReferenceEntry` and serialized beside `baselinePresent`; the release-readiness schema version does not change, since a new optional field is `AdditiveOptional` by the catalog's own stability rules.
- **`Drift.fs` needs no change.** It carries `.gitignore` presence-only for `doctor`/`upgrade`. The optional later `doctor` check for a workspace still carrying the ADR-0018-era rule is explicitly out of scope (ADR-0026, Consequences).
- **Two `.gitignore` adoptions, one decision.** The seeded consumer fragment and this repository's own dogfood rule are separate files with the same trap. ADR-0026 states both must land in #177 "or SDD will not dogfood the artifact it ships."
- **Existing golden fixtures will move.** `docs/release/release-readiness.json` and its test baseline are byte-locked and regenerate with the new entry and flag; a new `goldens/readiness/ship-verdict.json` joins the readiness golden set. Those updates are the evidence the behavior changed.
- **No history rewrite, no cleanup.** Already-ignored verdicts in consuming repos are not retroactively committed, and nothing is swept. The change adds a committed artifact going forward.
